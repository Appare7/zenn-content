Claude Fable 5 の話題って「ベンチマークがすごい」ばかり先行しがちなんですが、日々エージェントを組んでいる側からすると、本当に効いてくるのは**プロンプトと足場（scaffolding：モデルを動かす周辺の仕組み）の作法がガラッと変わった**という点なんです。旧モデル（Opus 4.8 など）の感覚のまま投げると、むしろ品質が落ちることすらあります。

今回は Anthropic 公式の [Prompting Claude Fable 5](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/prompting-claude-fable-5) を軸に、「今日から効く勘所」だけをぎゅっと絞って紹介します。入門記事ではないので、Claude Code や API でエージェントを組んでいる人向け。そのぶん「知らないとハマる」話を遠慮なく詰め込みました。

![Claude Fable 5 エンジニア向けガイド](/images/claude-fable-5-engineer-tips/figure.png)

## 1. 効きを回すのは `budget` じゃなく `effort`

まず最初に言っておきたいのが、ネットでよく見かける「Fable 5 は `budget_tokens` を 4000〜6000 に設定して…」という類の解説。あれ、そのまま真に受けないほうがいいです。公式ドキュメントいわく、Fable 5 は **adaptive thinking（適応的思考）のみ**で、**拡張思考の予算指定（budget_tokens）はもう廃止**されているんです。

じゃあ何で調整するのかというと、主役は **`effort`（努力度）パラメータ**。知能・レイテンシ（応答の遅さ）・コストのトレードオフは、ぜんぶここで回します。

![effort の段階と使い分け](/images/claude-fable-5-engineer-tips/figure2.png)

- 迷ったら **`high`**。既定はこれでOK。
- 精度が命の仕事だけ **`xhigh`**。
- 定型作業や、サクサク対話したいときは **`medium` / `low`**。

面白いのが、**Fable 5 の低 effort でも、旧モデルの xhigh を上回ることが多い**という点。むやみに上げなくていいんです。「終わるけど遅いなあ」と感じたら、まず effort を下げてみてください。

あと、高 effort だと頼んでもいない整形やリファクタまで踏み込みがちなんですよね。抑えたいときは、禁止事項をズラズラ並べるより、短く一言添えるだけで効きます（公式の例を要約）。

> バグ修正に周辺の掃除は要らない。ワンショットの処理にヘルパーは要らない。仮の将来要件のために設計しない。動く最小限をやって。

## 2. 旧モデル向けの指示は、むしろ「消す」

ここ、いちばん見落とされがちな移行ポイントです。Fable 5 は**指示追従（instruction following）がかなり強くなった**ので、挙動をひとつずつ列挙しなくても、短い一言でだいたい steer（誘導）できちゃいます。

裏を返すと、**旧モデル向けに盛りに盛った長〜いプロンプトや Skill は、Fable 5 だとむしろ足を引っ張ることがある**んです。公式もはっきり「古い指示は見直して、既定挙動で足りるなら削れ」と書いています。

![旧モデル向けの過剰な指示はむしろ品質を下げる](/images/claude-fable-5-engineer-tips/figure3.png)

たとえば「冗長に説明しない・箇条書きにしすぎない・PR説明を盛りすぎない…」と個別に並べる代わりに、こんな一文で同じ効果が出ます。

> まず結論から。最初の一文は「何が起きたか／何が分かったか」に答えて。詳細と理由はその後。読みやすさと簡潔さは別物で、読みやすさを優先。

移行作業のコツは、既存の system プロンプトや Skill を**「引き算」の目で見直す**こと。足すより削るほうが効くフェーズに入ったんです。

## 3. 「思考を本文に書かせる」指示は地雷

これは知らないと確実にハマるやつ。Fable 5 には**安全分類器（危険な出力を判定するフィルタ）**が入っていて、そのひとつが `reasoning_extraction`（モデルの内部推論の抽出）というカテゴリなんです。

つまり、「考えた過程を本文に全部書き出して」「reasoning をレスポンスに転写して」みたいな **show-your-thinking 系の指示を入れっぱなしにしていると、この拒否カテゴリを踏んで、Opus 4.8 への自動フォールバックが増えます**。

![推論を本文に書かせる指示は拒否カテゴリを踏む](/images/claude-fable-5-engineer-tips/figure5.png)

推論を見たいなら、本文に書かせるんじゃなくて **adaptive thinking の `thinking` ブロックを読む**のが正しい作法。移行のタイミングで、既存の Skill / system プロンプトに「自分の思考を説明しろ」系の指示が紛れ込んでいないか、一度チェックしてみてください。地味〜に効きます。

## 4. 長時間ランを前提に足場を組み替える

Fable 5、1リクエストが**数分〜数時間**走ることがあります。長丁場のエージェント運用だと特に。ここでコード側の足場が旧来のままだと、普通に事故ります。

- **クライアントの timeout・streaming・進捗表示は先に直しておきましょう。** ブロッキングで待つのはやめて、スケジュールジョブなどで**非同期にチェック**する構成が安心です。
- **進捗の「捏造」対策**も要ります。長時間ランだと、実際のツール結果と食い違う進捗を報告してくることがあるんです。公式いわく、こう指示したらほぼ消えたとのこと。

> 進捗を報告する前に、各主張をこのセッションのツール結果に照らして確かめて。証拠を指せる仕事だけ報告して、未検証なら未検証と言って。

- **勝手なアクションの抑制**も一言で効きます。頼んでいないメール下書きや、防御的な git ブランチ作成をやってしまうことがあるので、境界を教えてあげましょう。

> ユーザーが問題を説明している・質問している・考えを口に出しているだけのときは、成果物は「あなたの所見」。所見を報告して止まって。頼まれるまで修正は当てないで。

## 5. 並列サブエージェント & メモリを積極的に

Fable 5 は**サブエージェントの派遣（delegation）が旧モデルよりずっと得意**になっています。せっかくなので使い倒しましょう。

- 独立したサブタスクは**どんどんサブエージェントに投げて**、本体は次を進めちゃう。オーケストレータとサブは**非同期通信**で。長寿命のサブはコンテキストを持ち越せるので、キャッシュ読みでコストも時間も浮きます。
- **検証は「自己批判」より「別のサブエージェント」が強い**んです。fresh context（まっさらな文脈）の検証役に、仕様と照らしてチェックさせるのが公式のおすすめ。
- **メモリを持たせると特に伸びます**。1教訓 = 1ファイル、先頭に要約1行。過去ランの学びを Markdown に貯めて参照させるだけでOK。

## 6. 使いどころ：Sonnet と Fable 5 の線引き

とはいえ、全部 Fable 5 で殴るのはお財布に優しくないんですよね。ざっくりの線引きはこうです。

![Sonnet と Fable 5 の使い分け](/images/claude-fable-5-engineer-tips/figure4.png)

- **単一ターンで終わる／5ステップ以下**の作業 → 軽いモデル（Sonnet など）で十分。
- **10ステップ以上を自律で回す／失敗の代償が大きい**仕事 → Fable 5 の出番。
- **model mixing**（推論の重い所だけ Fable 5、実行・整形は軽いモデル）でコストはだいぶ落とせます。

それと、評価するときの鉄則がひとつ。**「難易度レンジの"上"から試す」**こと。簡単なタスクだけで測ると Fable 5 を過小評価しちゃいます。人間なら数時間〜数日かかる仕事（大規模移行、複数段の分析、複雑なエージェントフロー）でこそ地力が出るタイプなんです。

## 7. 安全分類器を踏むことがある

最後に運用の注意をひとつ。Fable 5 の安全分類器は、**攻撃的サイバーセキュリティ・生物/生命科学・思考抽出**を見張っています。厄介なのは、**普通のセキュリティ実装みたいな良性の作業でも、稀に引っかかる**こと。

対策はシンプルで、**Opus 4.8 へのフォールバックを設定**して、`stop_reason: "refusal"` を握って自動で流すだけ。ここさえ組んでおけば、たまの拒否で運用が止まらずに済みます。

## まとめ

Fable 5 に乗り換えるときの合言葉は、**「賢くなった分、引き算しろ」**です。

- 思考量は `budget` じゃなく **`effort`**（budget_tokens は廃止）
- 旧モデル向けの過剰な指示・Skill は**むしろ削る**
- **show-your-thinking 系の指示は拒否を踏む**ので外す
- 長時間ラン前提で **timeout / 非同期 / 進捗の裏取り**を組み直す
- **並列サブエージェントと外部メモリ**をガンガン使う
- **Sonnet と Fable 5 を使い分け**、評価は難易度の上から
- **Opus 4.8 フォールバック**は必ず用意

「新モデルだから今までのプロンプトを盛ろう」じゃなくて、「賢くなったから減らせる」に発想を切り替える。これが Fable 5 を使い倒す一番の近道だと思います。
