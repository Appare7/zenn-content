2026年6月12日に Moonshot AI から、新しいコーディング特化のオープンウェイトモデル「Kimi K2.7 Code」が公開されました。この記事は特定の1記事をそのまま翻訳したものではなく、複数の海外記事と公式情報を突き合わせつつ、日本のエンジニア向けに読みやすく組み直したものです。ざっくり言うと、「オープンウェイトのコーディングAI、ついにクローズドAPIの代わりに“そのまま挿せる”ところまで来たかも？」という話なんです。

![Kimi K2.7 Code：1兆パラのコード特化オープンウェイトAI](/images/jp-digest-20260712-kimi-k2-7-code/figure1.png)

## Kimi K2.7 Code って何が出たの？

Kimi K2.7 Code は、Moonshot AI が2026年6月12日に公開したコーディング特化モデルです。Hugging Face で重みが配布されていて、ライセンスは改変MIT（Modified MIT）になっています。

ここでいうオープンウェイトは、「学習済みの重みを公開する方式」のことです。つまり、重みをダウンロードして自前環境で動かしたり、中身を検証したりできる、という“開き方”なんですね。もちろん、何でも完全に自由という意味ではなく、ライセンス条件はちゃんと見る必要があります。

位置づけとしては、一般チャット向けというより、コード“専用”の変種です。特に、エージェント型のコーディング、つまり複数ターンでツールを呼びながらコードを書いたり直したりする使い方に振った作りになっています。Claude Code や Cline のようなツールを普段から使っている人には、かなり気になる方向性ですよね。

![Kimi K2.7 Codeのスペック早わかり](/images/jp-digest-20260712-kimi-k2-7-code/figure2.png)

### スペックの要点

- 規模は総パラメータ1兆（1T）です。ただし、1トークンあたり実際に動くのは320億（32B）だけです。これは MoE（専門家混合＝必要な専門家だけ起こす方式）だからなんですね。
- エキスパートは384個あり、トークンごとに8個＋共有1個を使います。層数は61層です。
- 注意機構には MLA（Multi-head Latent Attention）が使われています。
- コンテキスト長は256Kトークンです。一度に読める分量がかなり大きいので、大きめのリポジトリを扱う用途でも期待しやすいです。
- INT4（4ビットに軽量化した量子化）が最初から同梱されて配布されています。
- MoonViT という4億（400M）パラメータの画像エンコーダを内蔵しています。つまり画像入力に対応していて、UIのスクショを見てコードを書く、みたいな使い方も視野に入ります。
- 前バージョン比で、考える途中の「思考トークン」を平均で約30%削減しています。同じ答えを、より少ない手数で出しやすくなった、というわけです。

1兆パラメータと聞くとかなり大きく感じますが、MoE（専門家混合）の構成なので、毎回ぜんぶが動くわけではありません。ここは実運用コストに効いてくるポイントです。ただし、ローカル運用の話になるとまた別で、そこは後半で触れます。

## 数字で見る：K2.6からどれだけ伸びた？

まずは、前バージョンの Kimi K2.6 との比較です。以下はすべて Moonshot 自社ベンチの数字です。この点はかなり大事なので、次のセクションでちゃんと釘を刺します。

- Kimi Code Bench v2：50.9 → 62.0（+21.8%）
- Program Bench：48.3 → 53.6（+11.0%）
- MLS Bench Lite（多言語コード）：26.7 → 35.1（+31.5%）
- MCP Mark Verified（自律的な道具使い）：72.8 → 81.1（+11.4%）
- MCP Atlas：69.4 → 76.0（+9.5%）
- Kimi Claw 24/7 Bench：42.9 → 46.9（+9.3%）

パッと見て目立つのは、Kimi Code Bench v2 の +21.8% と、MLS Bench Lite（多言語コード）の +31.5% です。コードそのものの能力や、多言語コードへの対応で伸びているように見えます。

もうひとつ注目したいのが、MCP（外部ツール接続の共通規格）系のベンチです。MCP Mark Verified が +11.4%、MCP Atlas が +9.5% 伸びています。ここが伸びているということは、ただコードを1発で出すだけではなく、ツールを呼びながら自律的に作業するエージェント用途をかなり意識して鍛えた形が見えるんですよね。

最近のコーディングAIは、「関数を書いて」だけではなく、「リポジトリを読んで、修正して、テストして、必要ならまた直して」までやらせる流れになっています。Kimi K2.7 Code も、まさにそちら側に寄せたモデルとして見るのが自然そうです。

## ここが大事：数字は全部「自社ベンチ」

ここはかなり大事です。上で出したスコアは、全部 Moonshot 自身のベンチマークです。Kimi Code Bench v2、Program Bench、MLS Bench Lite、MCP Atlas、MCP Mark Verified などですね。

なので、「K2.6 より K2.7 Code が良くなっている」という見方はかなりしやすいです。少なくとも、同じ会社の同じ系列のベンチで伸びているわけなので、前バージョンからの改善としては受け取りやすいですよね。

一方で、「他社モデルより上」とまで額面通りに受け取るのは、まだ少し早いです。公開時点では、SWE-bench Verified、Terminal-Bench、LiveCodeBench といった業界標準の共通ベンチで、第三者による独立検証の数字がまだ出ていません。

また、他モデルとの比較レビューも一部ありますが、モデルごとに実行環境が違うケースもあります。たとえば CLI（コマンドライン操作）や UI（画面操作）が違うと、同じモデル比較に見えても条件が完全にはそろっていないことがあるんです。

なので、ここでの一番安全な結論はシンプルです。導入前に、自分の手元の実タスクで試してみてください。ベンチの数字は参考になりますが、普段触っているリポジトリでどう動くかが、結局いちばん大事です。

## 使い勝手のクセ：思考モードと出力の固定

Kimi K2.7 Code は、性能や価格の面でかなり気になるモデルですが、使い勝手にはクセもあります。

まず、思考モード、つまり考えてから答える動作はオフにできません。ということは、考えるための思考トークンぶんも毎回課金されます。前バージョン比で思考トークンは平均約30%削減されていますが、それでも「思考なしで軽く返してほしい」みたいな使い方はしにくい、ということですね。

もうひとつ、温度（temperature）や top_p などのサンプリング設定が固定気味です。サンプリング設定は、ざっくり言うと出力のゆらぎ方を決める設定です。ここが自由に触りにくいので、「まったく同じ出力を毎回きっちり再現したい」という決定論的な使い方には向きにくい場面があります。

このあたりは、「安くて賢いけど、ちょっと融通が利かない場面もあるよ」という感じで見ておくのがよさそうです。エージェント用途では多少のゆらぎが問題にならないことも多いですが、生成結果の完全再現性が必要なワークフローでは注意したいところです。

## 料金：強みはやっぱり「安さ」

Kimi K2.7 Code の大きな強みは、やっぱり料金です。API料金は100万トークンあたり、入力が $0.95、キャッシュヒット時の入力が $0.19、出力が $4.00 です。

さらに、前バージョン比で思考トークンが平均約30%減っているので、実タスクあたりの実費も下がりやすくなっています。エージェント型のコーディングでは、モデルが何度も考えて、ツールを呼び、また考えて……という動きになりがちなので、この差はじわじわ効いてきます。

あるレビューの実測比較では、同じ処理をクローズド大手のモデルでやるより、実費が1/5〜1/7ほどに収まったという報告もあります。ただし、これは一次のレビュー記事における実測値であり、条件次第で変わります。タスク内容、プロンプト、実行環境、ツールの使わせ方でコストは変わるので、そこは正直に見ておきたいですね。

とはいえ、大量にエージェントを回す用途ほど、この価格差は効いてきます。たとえば日常的にコード修正、テスト追加、リファクタ、調査系タスクをエージェントに投げているチームなら、モデル単価の差がそのまま運用コストに響いてきます。

## 現場でどう使う？既存ツールにそのまま挿す

Kimi K2.7 Code が面白いのは、既存ツールにそのまま挿しやすいところです。API は OpenAI 互換／Anthropic 互換の両方のエンドポイント（接続先）を持っています。なので、base_url と APIキーを差し替えるだけで、いつものコードから呼べます。

OpenAI 互換の呼び出しイメージは、こんな感じです。

```python
from openai import OpenAI
client = OpenAI(api_key="sk-...", base_url="https://api.moonshot.ai/v1")
resp = client.chat.completions.create(
    model="kimi-k2.7-code",
    messages=[{"role": "user", "content": "このモジュールをリファクタしてテストも足して"}],
)
```

この「互換エンドポイントがある」というのは、現場ではかなりありがたいポイントです。すでに社内ツールや検証コードが OpenAI 互換のクライアント前提で書かれているなら、差し替え検証のハードルがぐっと下がります。

さらに Anthropic 互換のおかげで、Claude Code、Cline、Roo Code といったエージェントの「中身」を Kimi に差し替えて動かせます。環境変数の設定でOKなので、普段使っているワークフローを大きく崩さずに試せるのがいいところです。

一方で、ローカルで自前ホストする場合は少し話が変わります。vLLM、SGLang、KTransformers 等での自前ホストは可能です。ただし、1兆パラメータの MoE なので、完全精度〜量子化でだいたい600GB級のメモリが要ります。個人PC、たとえば RTX 4090 の24GBのような環境では現実的ではなく、マルチGPUサーバ向きです。

また、Ollama や llama.cpp の公式ビルドは、公開時点ではまだありません。ローカルで気軽に動かすというより、まずは API で試して、必要があれば大型GPUサーバでの運用を検討する、くらいの順番がよさそうです。

## 結局、誰に向いてる？

Kimi K2.7 Code が向いていそうなのは、まず Claude Code などのエージェントを大量に回していて、API費用を抑えたい人です。コーディングエージェントはトークンをかなり使うので、入力 $0.95、キャッシュヒット時 $0.19、出力 $4.00 という価格はかなり気になりますよね。

それから、オープンウェイトで中身を検証したい人、自前で回したい人にも向いています。Hugging Face で重みが配布されているので、クローズドAPIだけに依存しない選択肢として見られます。画像入力を使いたい人にも、MoonViT の内蔵はうれしいポイントです。UIのスクショを見せてコードを書く、という方向の使い方ができるのは、フロントエンド周りやツール開発では便利そうです。

一方で、慎重になった方がいい場面もあります。決定論的な出力がどうしても必要な場合、温度や top_p などのサンプリング設定が固定気味な点は気になります。また、最高性能を業界標準ベンチで担保したい場合も、公開時点では第三者の独立検証がまだ出ていないので、判断を急がない方がよさそうです。

ローカル運用についても同じです。手元に大型GPUサーバが無いのに「オープンウェイトだから自分のPCで動かそう」と考えると、かなり厳しいです。だいたい600GB級のメモリが必要になるので、個人PCで気軽に、というモデルではありません。

まとめると、まずは API で、自分のリポジトリの実タスクにぶつけてみるのが一番です。ベンチの数字も参考にはなりますが、最後に答えをくれるのは、自分のコードでの手応えなんですよね。リファクタ、テスト追加、バグ修正、調査タスクあたりから試してみると、向き不向きがかなり見えやすいはずです。
