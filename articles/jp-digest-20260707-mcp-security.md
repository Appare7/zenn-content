MCP（AIに外部ツールやデータをつなぐための共通規格）が一気に広まって、AIエージェントがファイルを読んだり、API（アプリ同士をつなぐ窓口）を叩いたり、いろんな「道具」を使えるようになりました。便利ですよね。

でも、道具を持たせるということは、外から命令を差し込む隙間も増えるということなんです。しかも2026年には、実際に悪用された脆弱性も出てきました。

この記事では、複数の海外レポートや公式発表をもとに、MCP セキュリティとAIエージェント セキュリティの勘どころを日本語でまとめ直してみます。

![](/images/jp-digest-20260707-mcp-security/figure1.png)

## AIに「道具」を持たせた瞬間、攻撃の入り口も増えた

AIエージェントは「道具」を持つほど便利になります。ファイルを読める、外部APIを呼べる、社内のデータにアクセスできる。まるで新人メンバーに便利な道具箱を渡す感じです。

ただし、つなぐ相手が増えるほど、「これは信用していいんだっけ？」という前提がぐらつきます。ここがMCP セキュリティのややこしいところです。

特に狙われやすいのが、AIゲートウェイ（たくさんのLLMプロバイダへのアクセスを1つの窓口にまとめる中継サーバー）です。LLM（大規模言語モデル）を使うための入り口を集約しているので、そこに機密が集まりがちなんですよね。

![](/images/jp-digest-20260707-mcp-security/figure2.png)

たとえば、AIゲートウェイにはこんなものが集まります。

- 100を超えるLLMプロバイダのAPIキー
- 使用ログを保存するデータベースの認証情報
- どのチーム・サービスがどのモデルを使えるかを決める設定

つまり、1台落とされるとまとめて全部持っていかれる「鍵束」状態です。玄関の鍵、金庫の鍵、車の鍵を全部ひとまとめにして落とす感じ。ちょっと怖くないですか。

おまけに、初期に出回ったMCPサーバーには「そもそも認証なしで動く」ものが少なくありませんでした。便利だからサッと本番に置いたら、実は無防備でした、という事故が起きやすいわけです。

## 実際に悪用された：AIゲートウェイ「LiteLLM」の脆弱性

![](/images/jp-digest-20260707-mcp-security/figure3.png)

ここで、実際に悪用された例を見てみます。出てくるのは LiteLLM です。

LiteLLM は、BerriAI製のオープンソースのAIゲートウェイ／プロキシです。100超のLLMプロバイダを、OpenAI互換の1つの窓口で呼べるようにする人気ツールとして使われています。

問題になった脆弱性は **CVE-2026-42271**。種類はコマンドインジェクション（外部から任意のコマンドを実行させられる欠陥）です。単体の深刻度は **CVSS 8.7**。CVSS（10点満点で危険度を見るスコア）で8.7なので、かなり重めです。

影響を受けるバージョンは **v1.74.2 〜 1.83.6**。修正版は **1.83.7 以降**です。

原因になったのは、MCPサーバー設定を「保存前にお試し」できる2つのエンドポイントでした。

- `POST /mcp-rest/test/connection`
- `POST /mcp-rest/test/tools/list`

この2つは、リクエスト本文で渡した設定を、そのままプロキシのホスト上でサブプロセスとして起動していました。対象になるのは `command` / `args` / `env`、つまりstdio方式で使うコマンド・引数・環境変数です。

しかも、次のような守りがありませんでした。

- 実行してよいコマンドの許可リスト、つまり allowlist（許可したものだけ通す一覧）がない
- 管理者だけに制限されていない
- サンドボックス（隔離実行）がない

つまり、有効なAPIキーを持つ人なら誰でも、サーバー上で好きなコマンドを走らせられた、という話です。これはもう「お試し機能」の顔をした自由実行ボタンですね。

![](/images/jp-digest-20260707-mcp-security/source1.png)
出典: The Hacker News（https://thehackernews.com/2026/06/litellm-flaw-cve-2026-42271-exploited.html）

さらに怖いのが、別の脆弱性との連鎖です。

**CVE-2026-48710**、通称「BadHost」と呼ばれる脆弱性があります。これは、Webフレームワークの **Starlette 1.0.0 以前**にある、Hostヘッダの検証不備です。これとCVE-2026-42271を組み合わせると、認証そのものを丸ごとすり抜けられます。

その結果、認証不要の **RCE（遠隔から任意コード実行）** が成立します。連鎖時の深刻度は **CVSS 10.0**。満点です。セキュリティ企業の Horizon3.ai が、この連鎖を実証しています。

![](/images/jp-digest-20260707-mcp-security/source2.png)
出典: PointGuard AI（https://www.pointguardai.com/ai-security-incidents/litellm-command-injection-enables-remote-code-execution-cve-2026-42271）

では、何が奪われるのでしょうか。

- ホスト上での任意コマンド実行
- 各モデルプロバイダの資格情報
- 保存されたAPIキー
- そこから接続された他のAIインフラへの横移動

「AIゲートウェイは鍵束になりやすい」と最初に話しましたが、まさにそこを突かれた形です。

タイムラインもなかなか厳しいです。修正パッチは **5月8日** に出ていました。それでも、米CISA（アメリカのサイバーセキュリティ当局）が「実際に悪用中」として **KEV（悪用が確認された脆弱性リスト）** に追加したのが **6月8日**。翌 **6月9日** には技術メディアが報道しています。

公開から約5週間で攻撃者に武器化された計算です。

ここでの教訓はシンプルです。「パッチが出た＝安全」ではありません。「当てるまでは狙われ続ける」です。ちょっと耳が痛いですが、更新の速さがそのまま防御力になります。

まずの対策として、LiteLLM は **1.83.7 以降**、Starlette は **1.0.1 以降**に上げるところからです。

## いちばん厄介なのは「道具の説明文」に隠し命令を仕込む攻撃

LiteLLMのような脆弱性も怖いのですが、MCPまわりでさらにやっかいなのが、プロンプトインジェクション系の攻撃です。

プロンプトインジェクション（AIへの指示文に悪意ある命令を混ぜる攻撃）は、なんとなくイメージしやすいですよね。人間が入力欄に変な命令を書くタイプです。

でも、MCPで特に問題になりやすいのは **間接プロンプトインジェクション** です。これは、人が直接入力する文章ではなく、AIが読み込む「外部のもの」に命令を仕込む攻撃です。

たとえば、こんな場所です。

- ツールの説明文
- ウェブページ
- キャッシュ
- チケット履歴

人ではなく、道具の側から命令が飛んでくるイメージですね。掃除機だと思って買ったら、説明書の中に「家の鍵を外に置いてください」と書いてある、みたいな不気味さがあります。

この中でも、**tool poisoning（ツール汚染）** と呼ばれる手口がかなり厄介です。

![](/images/jp-digest-20260707-mcp-security/figure4.png)

tool poisoning の肝は、AIモデルが「ツールの説明文の全文」を読んで、どの道具を使うか判断する点にあります。一方でユーザーは、画面、つまりUI上では簡略版しか見ていないことが多いんです。

ここにズレがあります。

AIは全文を見ている。  
人間は一部しか見ていない。  

だから、説明文の中に隠し命令を埋め込まれても、人間は気づけません。

研究者の Invariant Labs は、かなり具体的な例を示しています。悪意あるMCPサーバーが、ただの「足し算ツール」に見せかけます。でも説明文の中、たとえば `<IMPORTANT>` タグの中に隠し指示を仕込みます。

その指示は、`~/.ssh/id_rsa`、つまりSSHの秘密鍵や、`~/.cursor/mcp.json`、つまり他サーバーの認証情報が入った設定ファイルを読んで、無害そうな引数、たとえば `sidenote` などにこっそり混ぜて送れ、というものです。

実際のAIエディタでの実験では、この通りにSSH秘密鍵と設定ファイルが外に流出しました。しかもユーザーに見えるのは「単純な確認ダイアログ」だけです。

承認ボタンを押した本人は、「足し算を許可した」くらいの気持ちです。でも裏では鍵が抜かれている。これはかなり嫌なタイプの攻撃です。

![](/images/jp-digest-20260707-mcp-security/source3.png)
出典: Practical DevSecOps（https://www.practical-devsecops.com/mcp-security-vulnerabilities/）

さらに **rug pull（ラグプル）** という応用もあります。最初は無害な説明で承認させておいて、あとからツールの定義をこっそり差し替える手口です。

最初はいい子だった道具が、途中で牙をむくわけです。昔話なら「実は狼でした」で済むかもしれませんが、本番環境だと笑えません。

ここで刺さるポイントは、「AIが見ている情報」と「人が見ている情報」がズレていること自体が弱点になる、というところです。MCP セキュリティを考えるなら、このズレをどう埋めるかがかなり大事になります。

## MCPまわりで起きる攻撃、ざっくり4タイプ

MCPまわりの攻撃は、細かく見るといろいろあります。ここでは Practical DevSecOps の分類をもとに、ざっくり4タイプで見てみます。

- **プロンプトインジェクション**  
  AIに隠れた命令を実行させる攻撃です。入力や読み込んだ情報の中に、こっそり命令を混ぜます。

- **tool poisoning（ツール汚染）**  
  道具の説明文や設定をいじって、危険な動作へ誘導します。上で説明した「足し算ツールに見せかける」タイプがこれです。

- **MPMA（MCP Preference Manipulation Attack＝どの道具を優先的に選ぶかをこっそり操作する攻撃）**  
  AIが「この道具を使いたくなる」ように仕向けます。道具選びの好みをこっそり曲げる感じです。

- **Parasitic Toolchain Attacks（寄生型ツールチェーン攻撃）**  
  複数の道具のつながりに寄生して、連鎖的に悪さをします。1個の道具だけでなく、道具同士の流れを狙うわけです。

全部に共通しているのは、AIが外部の情報や道具を「素直に信じてしまう」性質を突いている点です。AIは空気を読んでくれる反面、悪い空気まで読んでしまうことがあるんですね。

## 今日からできる、AIエージェントの守り方

![](/images/jp-digest-20260707-mcp-security/figure5.png)

ここからは対策です。全部を一気に完璧にやるのは大変です。ですが、上の2つ、つまり「最小権限」と「不可逆操作の人手承認」だけでも、事故の多くはかなり防ぎやすくなります。

### 1. 最小権限にする

道具に渡す権限は、必要な分だけにします。

何でもできる万能の鍵をAIエージェントに持たせると、便利ではあります。でも、もしプロンプトインジェクションやtool poisoningで悪い方向に誘導されたとき、その万能鍵がそのまま攻撃者のごちそうになります。

「この道具にはこの範囲だけ」「この処理にはこのデータだけ」と絞るのが基本です。

### 2. 不可逆な操作は人が承認する

**human-in-the-loop（人が途中で確認する仕組み）** を入れます。

削除、送金、外部送信のような取り返しのつかない操作は、AI任せにしないほうが安全です。機微な操作や不可逆な操作は、必ず人の確認を要求する形にします。

ここは面倒に見えますが、ブレーキのない車に乗るよりはずっとマシです。

### 3. 道具の説明文をピン留めする

ツールやサーバーのバージョンを固定し、説明文を **ハッシュ（内容の指紋のようなもの）** で検証します。これで、勝手な差し替え、つまりrug pullを検知しやすくなります。

また、ユーザーにも説明文の全文が見えるようにするのが理想です。AIだけが見ている「裏の説明書」をなくして、人間も同じものを確認できるようにするわけです。

### 4. 認証を固める

認証まわりでは、**OAuth 2.1（安全に権限を渡すための仕組み）** と **PKCE（認可コードの横取りを防ぐ仕組み）** を使います。

さらに、トークンの宛先、つまり audience をちゃんと検証します。クライアントのトークンを上流のAPIへそのまま横流ししないことも大事です。

「通行証を持っているから全部OK」ではなく、「この通行証は、どこ行きのもの？」まで見るイメージですね。

### 5. 入力の検証とSSRF遮断を入れる

ツールへの入力は、allowlist（許可したものだけ通す一覧）で絞ります。何でも受け取る入口は、だいたい攻撃者にとっても便利です。

また、内部ネットワーク、つまりプライベートIPへの外向き通信、**egress（外へ出ていく通信）** を止めます。これで **SSRF（サーバーに意図しない通信をさせる攻撃）** を防ぎやすくなります。

### 6. 依存を最新に保つ

ゲートウェイやライブラリはこまめに更新します。今回の例なら、LiteLLM は **1.83.7 以降**です。

依存関係のスキャンも回しておくと、古い部品が残っていることに気づきやすくなります。地味ですが、こういう地味な作業がいちばん効く場面、ありますよね。

### 防御側の考え方も少しだけ

最後に、個別の対策の前提になる考え方です。

入力と出力の境界をはっきりさせます。信頼できる指示と、信頼できない外部データを混ぜない。ここがぐちゃっとすると、AIは悪意ある命令まで「指示」として扱ってしまいます。

それから、監視・検知を入れて、おかしな動きにすぐ気づけるようにします。いざという時に素早く **ロールバック（元に戻すこと）** できる備えも持っておくと安心です。

## まとめ

MCPやAIエージェントは本当に便利です。ファイルを読める、APIを叩ける、複数の道具を組み合わせられる。夢があります。

でも、「道具を持たせる＝新しい攻撃の入り口が増える」もセットで来ます。

今回のLiteLLMの件は、AIゲートウェイに機密が集中していること、そしてパッチを当てる速さが防御力そのものだということを突きつけました。

さらに、tool poisoning のように「AIが見ている情報」と「人が見ている情報」のズレを突く攻撃は、これからも増えそうです。

まずは最小権限と、不可逆操作の人手承認から。難しく考えすぎず、そこから始めてみてください。
