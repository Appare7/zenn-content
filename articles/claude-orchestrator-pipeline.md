## きっかけ:「全部AIに任せる」は失敗する

個人開発で「コンテンツ生成から公開までを自動化したい」と思ったとき、最初に考えるのは「LLM（大規模言語モデル）に全部やらせる」案です。プロンプトを1つ投げれば文章ができて、勝手に公開される——理想的に聞こえます。

でも実際にやってみると、ここが崩れます。LLMは文章生成は得意ですが、「git に push する」「ファイルを正しいパスに保存する」「ログを1行追記する」といった**確定的（決まった手順どおりに必ず同じ結果になること）な処理**を任せると、たまに手順を飛ばしたり、パスを間違えたりします。不可逆な操作（公開・送信・削除）でこれが起きると、取り返しがつきません。

そこで役割を分けました。**判断はAI、実行は普通のコード**。この記事では、その「司令塔（オーケストレーター）」パターンを実体験ベースで紹介します。

## 役割分担:AIは「決める」、コードは「やる」

私が組んだ構成はシンプルです。

- **AI（司令塔）**: 何を書くか決める / 既存記事と被らない切り口を選ぶ / 本文を生成する / どのスラッグ（記事URLの末尾になる文字列）で保存するか決める
- **スクリプト（実行役）**: ファイル保存・git commit・push・ログ追記といった、失敗が許されない確定的処理

ポイントは、**不可逆な操作をAIの手元から取り上げる**ことです。push のような操作は、AIが直接コマンドを叩くのではなく、専用スクリプト経由でしか実行できないようにします。こうするとAIが暴走しても、スクリプトのガード（前提チェック）で止まります。

司令塔は何でも構いません。私は手元で動かせる Claude Code を使っていますが、考え方は「AIに判断させ、危ない実行はコードに閉じ込める」という一点に尽きます。

## 実例:公開スクリプトに「ガード」を仕込む

実際に使っている公開スクリプトの核心部分です（Zenn と GitHub を連携すると、リポジトリに push するだけで記事が公開できます）。対象ファイルが存在し、変更があるときだけ push します。

```js
// 指定スラッグの記事だけをコミットして push する
export function publish({ slug, message }) {
  const cwd = process.env.ZENN_REPO; // 記事リポジトリのローカルパス
  // ガード1: リポジトリが本当に存在するか
  if (!existsSync(path.join(cwd, ".git"))) {
    throw new Error(`git リポジトリじゃない: ${cwd}`);
  }
  const file = `articles/${slug}.md`;
  // ガード2: 記事ファイルが実在するか
  if (!existsSync(path.join(cwd, file))) {
    throw new Error(`記事が無い: ${file}`);
  }
  git(cwd, ["add", file]);
  // ガード3: 差分が無ければ何もしない（空コミット防止）
  if (!git(cwd, ["status", "--porcelain", file])) {
    console.log("変更なし。skip");
    return;
  }
  git(cwd, ["commit", "-m", message || `add article: ${slug}`]);
  git(cwd, ["push", "origin", "main"]);
}
```

3つのガードが効いていて、前提が崩れていると **push する前に必ず例外で止まります**。AIには「このスクリプトを呼べ」とだけ指示すればよく、git の細かい操作をAIに任せる必要がありません。

## 「下書きで出す」を初期値にする

公開系の自動化で一番怖いのは「いきなり本番公開」です。Zenn の GitHub 連携では、フロントマター（記事先頭の設定欄）の `published` を `false` にしておくと**下書き扱い**になります。GitHub にプッシュしても公開されず、自分でレビューしてから `true` に変える運用ができます。

```yaml
---
title: "記事タイトル"
emoji: "🪄"
type: "tech"
topics: ["claudecode", "ai"]
published: false   # まず下書き。レビュー後に手動で true へ
---
```

自動化は「最後の一歩（公開）だけ人間が押す」設計にすると、安心して回せます。push 自体は自動、公開判断だけ手動、という線引きです。

## やってみて分かったこと

- **AIに不可逆操作を直接やらせない**。push・送信・削除は、ガード付きのスクリプト経由に限定する
- **失敗したら止めてログを残す**。「とりあえず続行」より「原因を残して停止」のほうが、後から直しやすい
- **published: false を初期値に**。自動でも、公開の最終判断は人が持つ

なお、Zenn 側がプッシュを検知してデプロイするまでの所要時間は環境によって変わるため、ここでは「数分程度」とだけ書いておきます（断定しません）。

## まとめ

「AIに全部やらせる」のではなく、**判断はAI・実行は確定的なコード**に分けるのが、自動化パイプラインを安全に回すコツでした。AIを司令塔に、危ない操作はガード付きスクリプトに閉じ込める。この役割分担にしてから、自動化が「怖いもの」ではなく「任せられるもの」になりました。個人開発で公開系の自動化に踏み出すときの、最初の一歩として参考になればうれしいです。
