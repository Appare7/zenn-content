## そもそも「推論エンジン」って何をしてるの？

この記事は、複数の海外ベンチマーク記事や技術解説をもとに、日本のエンジニア向けに噛み砕いて再構成したものです。1本の記事の翻訳ではなく、「結局、自分は何を選べばいいの？」に寄せて読めるようにしています。

最初に言いたいことはシンプルです。**「どのモデルを使うか」と同じくらい、「何で動かすか」が効きます。**  
同じLLM（大規模言語モデル）、同じGPU（画像処理向けの計算装置）でも、推論エンジン次第で速度が10倍以上変わることがあるんです。早すぎません？

推論エンジン（inference engine＝学習済みモデルを実際に動かして文章を生成する土台）は、モデル本体、つまり「重み」と呼ばれるファイルを実際に動かすための仕組みです。モデルファイルだけ置いてあっても、それはまだ“材料”に近いんですよね。  
それを読み込んで、文章を生成して、複数人からのリクエストをさばいて、なるべくGPUを遊ばせないようにする。ここを担当するのが推論エンジンです。

効いてくる代表的な仕組みは、ざっくり3つあります。

1つ目は、**連続バッチ処理（continuous batching＝終わったリクエストを抜いて、次を隙間に差し込む方式）**です。  
素朴に1件ずつ処理すると、GPUが待ち時間だらけになります。レジで1人ずつ完全に会計が終わるまで次の人を入れない、みたいな感じですね。連続バッチ処理では、終わった人の枠に次の人をどんどん入れていくので、GPUを常に**85〜92%**働かせ続けられます。

2つ目は、**PagedAttention（ページド・アテンション）**です。これは、計算の途中結果である**KVキャッシュ（計算の途中結果）**を、大きな連続領域ではなく、小さなブロックに分けて管理する仕組みです。  
メモリって、大きな机を一気に確保しようとすると、空きはあるのに形が合わなくて使えないことがありますよね。PagedAttentionは、小さな付箋をうまく並べるようにメモリを使うので、ムダを**19〜27%**削減できます。

3つ目は、**量子化（数値を軽くする圧縮）**です。GGUF（量子化モデル形式）やGPTQ（量子化方式）、FP4（4ビット浮動小数点）などが出てきます。数値の表現を軽くすることで、同じGPUにより大きなモデルを載せられたり、速度を稼げたりします。

つまり、推論エンジンは「ただモデルを起動する道具」ではありません。  
**同じGPU・同じモデルでも、この土台の作りで速度が段違いになる**。ここが今回の記事の芯です。

## 顔ぶれは5つ。まず「ローカル勢」と「本番勢」で分けると分かりやすい

今回くらべるのは、**vLLM / SGLang / TensorRT-LLM / Ollama / llama.cpp** の5つです。いきなり横並びで全部くらべると、かなり混乱します。なのでまずは、「ローカル勢」と「本番勢」の2レーンで見るのが分かりやすいです。

### ローカル勢：1人で、手元で、まず動かす

まずは **Ollama**。これは「とりあえずローカルでLLMを触りたい」人にとって、かなりありがたい存在です。  
1コマンドでモデルのダウンロード、量子化、起動までやってくれます。OpenAI互換のローカルAPI（外部サービス風に呼べる手元の窓口）も立てられますし、VRAM（GPUメモリ）の管理やCPU（中央処理装置）への切り替えも自動です。

「モデルを選んだ。で、どうやって動かすの？」という段階なら、Ollamaの気軽さは本当に強いです。ノートPCで試すなら、まずここからでいい場面が多いです。

もう1つが **llama.cpp**。こちらは余計な依存が少なく、NVIDIA、AMD、Apple Metal、CPUのみ、さらにはラズパイまで、ほぼどんなハードでも動くのが強みです。  
GGUFという量子化形式の標準的な土台でもあり、「とにかく軽く、広く、いろんな環境で動かしたい」なら頼りになります。

### 本番勢：チームで、大人数をさばく

一方で、本番APIとして複数人に使わせるなら、話が変わってきます。

**vLLM** は本番向けの定番です。PagedAttentionと連続バッチ処理でGPUを**85〜92%**稼働させます。量子化もFP8（8ビット浮動小数点）、FP4、INT8（8ビット整数）、INT4（4ビット整数）、GPTQ、AWQ（量子化方式）、GGUFなど、対応が広いです。

**SGLang** は、RAG（検索して答えを作る仕組み）やマルチターン会話（何往復も続く会話）に強いエンジンです。理由は後で触れますが、「同じ前置き」や「同じ資料」を何度も使う場面で効いてきます。

**TensorRT-LLM** はNVIDIA特化で、高トラフィックの本番で最大スループット（単位時間あたりに処理できる量）を狙うための選択肢です。かなりガチめですね。

ざっくり言うと、**まずOllamaで触ってみて、人が増えたらvLLM/SGLangへ**。この順番が王道です。最初から全部を本番構成にしようとすると、だいたい疲れます。

## 数字で見よう。速度差は本当に10倍以上あった

では、実際にどれくらい差が出るのか。ここは数字で見てみます。

紹介するのは、あるベンチマークの結果です。環境は **NVIDIA RTX PRO 6000 Blackwell、96GB VRAM、モデルはLlama 3.1 8B**。  
ただし、これはあくまで**1つの環境での結果**です。GPUやモデル、量子化方式が変われば数字は動きます。ここは正直に見ておきたいところです。

ピーク速度、つまり1秒あたりの生成トークン数はこうでした。トークンは文章を細かく分けた単位で、多いほど速く生成できているイメージです。

- vLLM（NVFP4という量子化）: **8,033 tok/s**（このベンチの最速）
- SGLang（GPTQ-INT4）: **6,395 tok/s**
- Ollama（Q4）: **484 tok/s**（専用サーバーのおよそ10分の1）

vLLMが8,033 tok/s、Ollamaが484 tok/s。かなり差がありますよね。もちろんOllamaが悪いという話ではありません。役割が違うんです。Ollamaは「気軽に動かす」ための道具で、vLLMやSGLangは「大量にさばく」ための道具です。

体感速度に効く、最初の1文字が返るまでの時間も見てみます。これは **TTFT（Time To First Token＝最初のトークンが返るまでの時間）** と呼ばれます。

- vLLM: **13.8ms**
- SGLang: **14.2ms**
- Ollama: **65ms**

vLLMはOllamaより、約6倍速く最初の1字を返しています。チャットUIだと、この差はけっこう体感に出ます。「あ、返ってきた」と感じるまでの間ですね。

さらに、128人が同時に叩いたときの結果もあります。  
**vLLMとSGLangは成功率100%**。一方で、**Ollamaはこの負荷で処理に失敗**しました。Ollamaは1件ずつ順番に処理する作りなので、多人数同時アクセスには弱いんです。

面白いのは、量子化を同じGPTQ-INT4でそろえてくらべると、**SGLangがvLLMを17%上回る**場面もあったことです。  
つまり、「vLLMが常に最速」「SGLangが常に上」という単純な話ではありません。量子化やハード、ワークロードでひっくり返ります。

## vLLM と SGLang、中身の違いは「メモリを使い回すか、捨てるか」

ここは、「vLLMとSGLang、結局どっち？」を分ける肝です。  
ざっくり言うと、**vLLMは終わったらきれいに捨てる**。**SGLangは使い回せるものを覚えておく**。この違いです。

vLLMのPagedAttentionは、KVキャッシュを小さなブロックで管理します。そして、リクエストが終わったら即解放します。毎回まっさらな状態で処理するので、どんなプロンプトにも公平です。  
いろんな人が、いろんな内容を、バラバラに投げてくるAPIには向いています。まさに本番の汎用サーバーという感じです。

一方で、SGLangには **RadixAttention（ラディックス・アテンション）** があります。これは「共通する前半部分」をキャッシュして使い回す仕組みです。  
たとえば、同じシステムプロンプトを毎回付けるチャットボット。あるいは、同じ資料を参照しながら答えるRAG。こういう場面では、毎回ゼロから読み直すより、前に使った部分を再利用できたほうが速いですよね。

別のH100（データセンター向けGPU）でのベンチ、モデルはLlama 3.1 8Bでは、**SGLangがvLLMより29%高いスループット**という報告もあります。  
さっきのBlackwellベンチとは数字が違います。これがまさに、環境で結果が変わる好例です。

さらに、複数の資料を共有するRAGでは**最大6.4倍**、DeepSeek系モデルでは**3.1倍**速いという結果もあります。ここまで来ると、単なる微差ではありません。

使い分けの目安はこうです。

- 毎回バラバラのプロンプトを大量にさばくなら **vLLM**
- 同じ前置き、RAG、長い会話が多いなら **SGLang**

ちなみに、vLLMはTPU（AI向け計算装置）やAMD/Intel系など対応ハードが広く、コントリビューター（開発参加者）の層も厚いです。SGLangは研究寄りで、学習コストはやや高めです。  
なので、チームで安定運用したいならvLLM、ワークロードがハマるならSGLang、という見方がしっくりきます。

## じゃあ Ollama と vLLM の違いは？ 「気軽さ」と「捌く力」のトレードオフ

ここがいちばん検索される疑問かもしれません。  
**OllamaとvLLM、どっちを使えばいいの？** というやつです。

答えはかなりシンプルです。  
**Ollamaは気軽さの王者、vLLMは捌く力の王者**。役割が違います。

Ollamaは、1コマンドで動きます。VRAM管理もCPUへの切り替えも自動です。ノートPCでサッとモデルを試したいなら最高です。  
「このモデル、日本語どれくらい話せるかな」「この量子化なら手元で動くかな」みたいな検証には、とても向いています。

ただし、Ollamaは1件ずつ順番に処理する作りです。複数人が同時に使うと、速度がガクッと落ちます。さっきの128並列で失敗した結果は、かなり象徴的です。  
1人で使っていると快適でも、チームに配った瞬間に「なんか遅くない？」となることがあります。ここ、知らないとハマるやつです。

vLLMはその逆です。セットアップはOllamaより手間がかかります。でも、連続バッチ処理でGPUを**85〜92%**働かせ、5〜100人が同時に叩く本番APIをちゃんと捌けます。  
社内チャットボット、プロダクト機能、複数ユーザー向けの推論API。こういう用途なら、vLLMの出番です。

llama.cppにも触れておきます。llama.cppはOllamaの中身でもある土台で、依存なし・CPUでも動く超軽量な存在です。Mac（Apple Metal）やCPUだけの環境、メモリを切り詰めたい単独作業なら、llama.cppやExLlamaV3も選択肢になります。

ひとことでまとめるなら、こうです。

- **1人で試す → Ollama**
- **本番でみんなに配る → vLLM**

ここを取り違えると、「同じモデルなのに遅い！」と悩みます。モデルのせいではなく、走らせ方の問題だった、というわけです。

## で、結局どれを選べばいい？（状況から逆算）

最後は、状況から逆算して選びましょう。名前の有名さで選ぶより、自分の使い方に合わせるほうが失敗しにくいです。

- ノートPCで1人、まず試したい → **Ollama**
- Mac・どんなハードでも・CPUでも動かしたい → **llama.cpp**
- 5〜100人が同時に使う本番API → **vLLM**
- RAG・長い会話・同じ前置きが多い → **SGLang**
- NVIDIAで極限までスループットを絞りたい → **TensorRT-LLM**

TensorRT-LLMについては、少しだけ注意があります。  
あるベンチでは、vLLMやSGLangの起動、つまりコールドスタート（初回起動）が約60秒だったのに対し、TensorRT-LLMはコンパイル込みで**約28分**かかりました。速さと引き換えに、準備がかなり重いんです。

しかも、差が開くのは「並列数がかなり大きくなってから」です。そこまでの規模でなければ、最初から無理にTensorRT-LLMを選ばなくても大丈夫です。  
「速そうだから最初からこれで！」と行くと、セットアップや運用の重さで遠回りになることがあります。

王道の順番は、かなり素直です。

1. まずローカルで1つ動かす  
2. 必要になってから本番エンジンへ移る  
3. さらに規模が大きくなったら、特化型も検討する

最初からTensorRT-LLMでガチガチに固めるのは、たいてい遠回りです。まずはOllamaやllama.cppで触って、ユーザーが増えてきたらvLLMやSGLangへ。これくらいの温度感で進めると、かなり楽です。

## まとめ：モデル選びと同じくらい「エンジン選び」が効く

LLMをローカルPCや自社サーバーで動かすとき、つい「どのモデルが賢いか」に目が行きます。もちろん、それは大事です。  
でも同じくらい、**何で動かすか**も効きます。

同じモデル・同じGPUでも、推論エンジンで速度は10倍以上変わることがあります。vLLMやSGLangのような本番向けエンジンは、連続バッチ処理やキャッシュ管理でGPUをしっかり使い切ります。Ollamaやllama.cppは、手元で気軽に動かすところに強みがあります。

なので、選び方はこうです。

- まず試したいなら **Ollama**
- どんな環境でも軽く動かしたいなら **llama.cpp**
- 本番APIで複数人に配るなら **vLLM**
- RAGや長い会話が多いなら **SGLang**
- NVIDIA環境で極限まで詰めるなら **TensorRT-LLM**

ただし、ベンチマークの数字は環境で動きます。GPU、モデル、量子化、同時アクセス数、RAGの有無で結果は変わります。  
最後は、自分のGPU、自分のモデル、自分の使い方で小さく試すのがいちばん確実です。

「モデルは決めた。じゃあ何で動かす？」  
ここまで考えられると、コストも体感速度もかなり良くなります。焦らず、まずは手元で動かしてみてください。そこから必要に応じて、本番向けのエンジンに育てていけば大丈夫です。
