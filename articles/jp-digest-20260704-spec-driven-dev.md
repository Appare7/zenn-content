「AIにお願いしたら、それっぽいコードが一瞬で出てきた！」——この感動、2025年にみんな一度は味わいましたよね。いわゆる**バイブコーディング（vibe coding）**、ノリでAIに書かせるやつです。

ところが2026年に入って、本番のプロダクトを作るプロたちの間で流れが変わってきました。キーワードは**仕様駆動開発（spec-driven development、しようくどうかいはつ）**。ざっくり言うと「AIに書かせる前に、何を作るかを文章でちゃんと決めておく」やり方です。

この記事は、海外の技術メディアや開発ツールの公式解説など**複数のソースを読み込んで再構成**したものです。バイブコーディングと仕様駆動開発は何が違って、どう使い分けるのか。道具は何があるのか。やさしく整理していきますね。

![仕様駆動開発とバイブコーディングを図解したアイキャッチ](/images/jp-digest-20260704-spec-driven-dev/figure1.png)

## そもそも「バイブコーディング」って何だった？

まず「vibe coding とは」から。この言葉、実は生まれたばかりなんです。2025年2月に研究者のAndrejKarpathy（アンドレイ・カルパシー）さんがSNSで言い出したのが最初。「コードの存在すら忘れて、ノリ（vibe）に身を任せる新しいコーディングがある」——そんなノリの投稿でした。

これが刺さって、英語圏の辞書が「今年の言葉」に選ぶほど広まりました。プロンプト（AIへの指示文）を書いて、返ってきたコードを深く読まずに受け入れて、動かなかったらまた指示する。この手軽さが、試作や個人開発で爆発的にウケたわけです。

![バイブコーディングから仕様駆動開発へ至る2年間のタイムライン](/images/jp-digest-20260704-spec-driven-dev/figure2.png)

面白いのは、言い出しっぺのKarpathyさん自身がもう次を見ていること。海外メディアいわく、賢くなったLLM（大規模言語モデル）のおかげで「去年のバイブコーディングはもう古い」とまで言われ始めています。たった2年で流行語が「作り方そのもの」を変えてしまった。この速さ、ちょっとすごくないですか？

![出典: The New Stack（https://thenewstack.io/vibe-coding-is-passe/）のトップ画面](/images/jp-digest-20260704-spec-driven-dev/source2.png)
*出典: The New Stack（https://thenewstack.io/vibe-coding-is-passe/）*

## なぜプロは「仕様駆動開発」に乗り換え始めたのか

バイブコーディング、最初はめちゃくちゃ速いんです。デモまでは。問題はそこから。

海外の解説で繰り返し出てくるのが**「3ヶ月の壁」**という言い回し。ノリで作り続けると、だいたい3ヶ月目あたりで技術的負債（あとで返すことになる"借金"のようなコードの乱れ）が膨らんで、チームの手が止まり始めるという話です。あるレポートでは、この段階でスプリント（開発の一区切り）の2〜3割がバグ修正に食われる、なんて数字も出ていました。

![コードの重複が増える「3ヶ月の壁」を示した数値図解](/images/jp-digest-20260704-spec-driven-dev/figure3.png)

裏付けっぽいデータもあります。約2.1億行のコードを調べた調査（GitClear）によると、2021年から2024年にかけて、コードを整理し直す**リファクタリングの割合が約6割も減り**、逆にコピペで貼られたコードが増えたとか。AIは既存の部品を探して使い回すより、その場で自己完結したコードを生成しがちなので、重複が最大8倍にふくらんだケースもあると報告されています。

もう一つ、耳が痛い話。ある実験では、開発者は「AIで2割くらい速くなった気がする」と感じていたのに、既存コードでの実測ではむしろ遅くなっていた、という結果も出ています。体感と実測がズレる。これ、けっこう怖いですよね。

こうした「速いけど積み上がる」問題に効くのが、仕様駆動開発というわけです。

## 仕様駆動開発とは？ Markdownで作る4ステップ

じゃあ「仕様駆動開発 とは」何なのか。合言葉は**「仕様（spec）が真実の源、コードはそこから生まれる生成物」**。チャットに思いつきを投げる代わりに、先に仕様をMarkdown（マークダウン、軽量な記法のテキスト）で書いておいて、AIエージェントにはそれを"契約書"として実装させる。この順番が逆になっているのがポイントです。

代表的なのが、次の4ステップ。GitHub Spec Kit というツールがこの流れを標準化して広めました。

![仕様駆動開発の4フェーズ Specify→Plan→Tasks→Implement のフロー図](/images/jp-digest-20260704-spec-driven-dev/figure4.png)

1. **Specify（要件）**：何を作るか、ユーザー目線で「何が欲しいか」を書く
2. **Plan（設計）**：どう作るか、アーキテクチャ（構造）や依存関係を先に決める
3. **Tasks（タスク）**：手順を依存順に並べた、こまかいチェックリストにする
4. **Implement（実装）**：AIが1ステップずつ実行し、人がレビューする

キモは、要件・設計・タスクが**ファイルとして残る**こと。しかも譲れないルール（たとえば「全テーブルに行レベルのアクセス制御を必ずかける」みたいな鉄則）は **Constitution（憲法）ファイル**に固定しておいて、どの仕様にも効かせられます。半年後に読み返しても、コードのすぐ横に「なぜこうなっているか」が残っている。これがノリ開発との一番の違いです。

## 数字と体感で見る、2つのやり方の違い

ここまでを表で並べてみますね。

![バイブコーディングと仕様駆動開発の比較表](/images/jp-digest-20260704-spec-driven-dev/figure5.png)

ざっくり言えば、**バイブコーディングは「最初のデモまで最速」、仕様駆動開発は「書き始めは遅いけど後半で伸びる」**。書き上げたあとは仕様駆動のほうが従来比で+80〜150%速い、なんて声もありますが、この手の数字はソースによって幅があるので「そういう報告もある」くらいに受け取っておくのが安全です。

大事なのは数字そのものより、**向いている場面が違う**という事実のほう。ここを取り違えると、試作に重い仕様を書いて時間を溶かしたり、本番システムをノリで作って3ヶ月後に泣いたり、ということになります。

## 現場の道具：Kiro・GitHub Spec Kit・BMAD・GSD

「仕様駆動開発 ツール」で調べると、いくつか定番が出てきます。発想は同じでも、味付けが違うんですよね。

![仕様駆動開発を支える代表的なツール4つの比較](/images/jp-digest-20260704-spec-driven-dev/figure6.png)

- **Kiro**：仕様駆動をコンセプトに据えた公式のIDE（統合開発環境）。要件→設計→タスクの3画面で進めるのが特徴で、ノリで書く「Vibeモード」と仕様から作る「Specモード」を行き来できます。「kiro 使い方」を調べる人が増えているのも納得。
- **GitHub Spec Kit**：オープンソースのCLI（コマンドラインツール）。スターは9万超で、30以上のAIエージェントに対応。前述の4ステップと Constitution ファイルはここ発祥です。
- **BMAD-METHOD**：役割の違う12以上のエージェントが分担して開発を進めるフレームワーク。特定のAIベンダーに縛られないのが強み。
- **GSD**：調査・計画・実行・検証を並列で回す軽量なフレームワーク。ローカルのモデルでも動かせます。

どれも根っこは**「仕様が主役、コードは生成物」**で共通しています。まずは Kiro か Spec Kit あたりから触ってみると、感覚がつかみやすいと思いますよ。

## で、結局どっちを使えばいいの？

ここが一番聞きたいところですよね。海外の識者のコメントを読むと、答えはだいたい一致していて——**「敵同士じゃなくて、補い合う関係」**なんです。

ある人は「バイブコーディングと仕様駆動開発は競合じゃなく、それぞれ役割が違う」と言い、別の人は「仕様駆動開発はバイブコーディングの自然な成熟形だ」と表現しています。個人的にいちばんしっくりきたのはこの言い回し：

> バイブコーディングは、そのアイデアが作る価値があるか確かめるのに向いている。仕様駆動開発は、価値があると分かったあとに"ちゃんと作る"ためのやり方だ。

つまり、**アイデア検証やハッカソン、社内ツールならバイブでサクッと。本番投入・チーム開発・長期保守なら仕様駆動でどっしり**。最初はノリで探って、これは本気で作るぞと決まった瞬間に仕様へ書き起こす——この乗り換えが一番おいしい使い分けだと思います。

## 仕様駆動開発にも落とし穴はある

とはいえ、仕様駆動開発は銀の弾丸ではありません。ちゃんと弱点もあります。

- **仕様が陳腐化する**：コードだけ直して仕様の更新をサボると、"嘘の設計図"が残ってかえって混乱します。
- **書きすぎ問題**：完璧な仕様を目指しすぎて、いつまでも実装に入れない「分析まひ」に陥りがち。
- **手間はやっぱり増える**：最初にきっちり書く分、立ち上がりは遅い。小さな試作にこれをやると、ただただ重いだけ。

だからこそ「全部を仕様駆動で」ではなく、**規模と寿命で選ぶ**のが現実解。使い捨ての実験にまで憲法ファイルを書く必要はない、というわけです。

## まとめ

- **バイブコーディング**＝ノリでAIに書かせる。試作・学習・個人ツールに最速。ただし3ヶ月あたりで技術的負債が積み上がりやすい。
- **仕様駆動開発**＝要件・設計・タスクを先にMarkdownで固め、AIはそれを契約書として実装。本番・チーム・長期保守で強い。
- 2つは敵ではなく**補完関係**。アイデア検証はバイブ、作り込みは仕様駆動、と乗り換えるのが賢い。
- 道具は **Kiro / GitHub Spec Kit / BMAD / GSD** あたりが定番。発想はどれも「仕様が真実の源」。

「AIにお任せ」の次は、「AIに何を作らせるかを、人がちゃんと言葉にする」時代。仕様駆動開発、名前は堅いですけど、やってることは案外シンプルです。次に何か本気で作るとき、いきなりチャットに投げる前に、要件を3行だけ書いてみる。そこから世界が変わるかもしれませんよ。
