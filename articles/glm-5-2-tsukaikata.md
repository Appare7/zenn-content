「Claude、めっちゃ便利。でも毎月の料金がじわじわ効いてくる…」——コーディングにAIを使い倒している人ほど、この悩みにぶつかりますよね。そんな中、2026年6月中旬に中国のZ.ai（Zhipu AI／智譜AI）が公開した **GLM-5.2** が「オープンモデルなのにClaude Opus級のコーディング性能を、数分の一のコストで出せる」と話題になっています。

この記事では、GLM-5.2 とは何かから、ベンチマーク、料金、そして本命の **Claude Code につないでコストを約1/6に抑える使い方** まで、はじめての人にもわかるようにやさしく順を追って解説します。Cline（クライン）や Cursor（カーソル）での設定、API を直接叩く方法までまるっとカバーするので、気軽についてきてください。

*巨大なモデル本体を、手元の開発ツールから呼び出して使うのが GLM-5.2 の基本スタイル。*

## GLM-5.2とは？Z.aiの1Mトークン・オープンモデルをざっくり把握

まず結論から。GLM-5.2 は、**コーディングとエージェント（自律的にタスクをこなすAI）用途に特化した大規模言語モデル（LLM）** です。ざっくり特徴を並べるとこんな感じ。

- **100万トークン（約1M）の長文コンテキスト**：大きなコードベースをまるごと読ませられる
- **オープンウェイト（重み公開）× MITライセンス**：自分のサーバーで動かすのも、商用利用も原則自由
- **コーディング性能がトップクラス**：後述のベンチマークで GPT-5.5 を上回り、Claude Opus 4.8 に肉薄
- **とにかく安い**：西側フロンティアモデル（最先端の商用モデル）の数分の一のコスト

「オープンウェイト」というのは、モデルの中身（学習済みの重みパラメータ）が配布されていて、誰でもダウンロードして動かせる状態のこと。ChatGPT や Claude のような「APIでしか使えない」クローズドなモデルとは対照的です。しかも GLM-5.2 は制約のゆるい **MITライセンス** なので、ダウンロードして自前で動かすのも、細かく調整（ファインチューニング）するのも、その上で商用サービスを作るのも基本OK。ここが人気の大きな理由です。

Zenn の読者的にいちばん美味しいのは、**この安くて賢いモデルを Claude Code や Cline のバックエンド（裏側で動く頭脳）に差し替えられる** という点。使い慣れたツールはそのままに、頭脳だけ格安のものに載せ替えるイメージですね。ではもう少し中身を見ていきましょう。

## GLM-5.2の特徴と技術仕様

### MoE（専門家混合）アーキテクチャ

GLM-5.2 は **MoE（Mixture-of-Experts＝専門家混合）** という設計を採用しています。Hugging Face の公式モデルカードによると、総パラメータ数は約 **7,530億（753B）**。ただし、これが毎回まるごと動くわけではありません。

MoE をたとえるなら、**大きな病院の受付** みたいなもの。患者さん（入力）が来るたびに全科の医師が総出で診るのではなく、症状に合った専門医だけが呼ばれますよね。MoE も同じで、入力ごとに「担当の専門家（エキスパート）」だけをピンポイントで起動します。だから総パラメータは巨大でも、1トークンあたりに実際に動く量はぐっと少なく、**賢さと計算コストのいいとこ取り** ができるわけです。

![MoE（専門家混合）のイメージ](https://raw.githubusercontent.com/Appare7/zenn-content/main/images/glm-5-2-tsukaikata/figure2.png)
*入力に応じて、必要な専門家（エキスパート）だけが起動する MoE の考え方。*

### 100万トークンの長文コンテキスト

GLM-5.2 の目玉が **100万トークンのコンテキストウィンドウ**（一度に読み込める文脈の量）です。トークンはAIが文章を処理する単位で、日本語だとおおよそ1文字＝1〜2トークンくらい。100万トークンあれば、**中規模のプロジェクトのソースコードを丸ごと** 放り込んでも余裕があります。

これがコーディングだと効きます。「このリポジトリ全体を踏まえて、複数ファイルにまたがる変更をして」といった、いわゆる長期タスク（long-horizon task）が得意。ファイルをちまちま切り貼りして文脈を与える手間が減るんです。

![100万トークンの長文コンテキストのイメージ](https://raw.githubusercontent.com/Appare7/zenn-content/main/images/glm-5-2-tsukaikata/figure3.png)
*大量のコードをまとめて飲み込めるのが、長文コンテキストの強み。*

### 思考努力レベル（High / Max）

GLM-5.2 には **思考努力レベル（thinking effort）** という概念があり、`High` と `Max` の2段階が用意されています。ざっくり言うと「どれだけじっくり考えてから答えるか」のダイヤルです。

- **High**：ほどよく速く、日常的なタスク向き
- **Max**：より深く推論する。難しいコーディングタスクではこちらが推奨

急ぎのちょっとした修正なら High、腰を据えて設計から任せたいときは Max、と使い分けるイメージですね。

### MITライセンスで重みを公開

前述のとおり GLM-5.2 の重みは **MITライセンス** で公開されています。実際 Hugging Face 上では `zai-org/GLM-5.2` として配布されており、公開から1か月ほどで月間15万ダウンロードを超える人気ぶり。ライセンスもしっかり「mit」と明記されています。

![Hugging FaceのGLM-5.2モデルページ](https://raw.githubusercontent.com/Appare7/zenn-content/main/images/glm-5-2-tsukaikata/source2.png)
*出典: Hugging Face（https://huggingface.co/zai-org/GLM-5.2）／ライセンス: MIT。753Bパラメータ、月間ダウンロード数などが確認できる。*

MITライセンスの何が嬉しいかというと、**「使うのも改造するのも商用に組み込むのも、ほぼ自由」** という点。自前のGPUサーバーで動かして社内ツールに組み込む、みたいな使い方も堂々とできます（現実的にはかなりのマシンパワーが要りますが…）。

## ベンチマーク：GPT-5.5を上回り、Claude Opus 4.8に肉薄

気になる性能を見ていきましょう。GLM-5.2 が話題になった最大の理由が、**コーディング系ベンチマークでの数字** です。

長期のコーディング能力を測る **FrontierSWE**（ソフトウェアエンジニアリング系のベンチマーク）で、GLM-5.2 は **74.4%** をマーク。これは、

- Claude Opus 4.8：75.1%（あと一歩！）
- **GLM-5.2：74.4%**
- GPT-5.5：72.6%

という並びで、**GPT-5.5 を上回り、トップの Claude Opus 4.8 にわずか0.7ポイント差まで迫った** という結果です。海外メディアの VentureBeat も「オープンウェイトの GLM-5.2 が、複数の長期コーディングベンチで GPT-5.5 を1/6のコストで上回った」と報じています。

![VentureBeatのGLM-5.2報道](https://raw.githubusercontent.com/Appare7/zenn-content/main/images/glm-5-2-tsukaikata/source3.png)
*出典: VentureBeat（https://venturebeat.com/technology/z-ais-open-weights-glm-5-2-beats-gpt-5-5-on-multiple-long-horizon-coding-benchmarks-for-1-6th-the-cost）*

もちろんベンチマークは万能の指標ではないので、「Claude Opus とまったく同じ体験ができる」とまで言い切るのは早計です。とはいえ、**オープンモデルがここまで商用トップに肉薄してきた** こと自体が、少し前なら考えられなかった快挙。しかもそれをこの価格でやってくる、というのが次の料金の話につながります。

## GLM-5.2の料金体系

GLM-5.2 の使い方は大きく2ルートあり、料金体系もそれぞれ違います。**定額のサブスク（GLM Coding Plan）** と、**従量課金のAPI** です。

![コストと性能のバランスのイメージ](https://raw.githubusercontent.com/Appare7/zenn-content/main/images/glm-5-2-tsukaikata/figure5.png)
*性能はフロンティア級に近づけつつ、コストは大きく下げる——それが GLM-5.2 の売り。*

### GLM Coding Plan（Lite / Pro / Max / Team）

Claude Code などでガッツリ使うなら、コーディング特化の定額プラン **GLM Coding Plan** が中心になります。ティア（等級）は複数あり、公表されている標準価格の目安はこんな感じ。

| プラン | 月額の目安 | ざっくりの位置づけ |
| --- | --- | --- |
| Lite | 約 $18〜 | まず試したい個人向け |
| Pro | 約 $72 | 日常的にヘビーに使う人向け |
| Max | 約 $160 | 大規模・チームのパワーユーザー向け |
| Team | 要確認 | チーム利用向け |

注目は **月18ドル（Liteプラン）から始められる** こと。Claude や ChatGPT のコーディング向けプランと比べても、かなり戦略的な価格です。「まずは安いプランで、自分の使い方に合うか試す」というのがやりやすい。

### 従量課金API・OpenRouter経由

もう一つが、使った分だけ払う **従量課金API**。Z.ai 公式APIの料金は、おおよそ **入力 100万トークンあたり $1.40、出力 $4.40**、キャッシュ済み入力なら約 $0.26 と案内されています。

さらに、複数のモデルを一つの窓口から呼べる中継サービス **OpenRouter（オープンルーター）** 経由でも使えます。OpenRouter では複数の提供事業者が同じ GLM-5.2 をホストしていて、価格は事業者ごとに幅がありますが、記事執筆時点では **入力 $0.93／出力 $3.00 前後** から選べる提供元もあります。GIGAZINE によれば、公開直後から OpenRouter 上での利用者が急増したとのこと。

![OpenRouterのGLM-5.2ページ](https://raw.githubusercontent.com/Appare7/zenn-content/main/images/glm-5-2-tsukaikata/source.png)
*出典: OpenRouter（https://openrouter.ai/z-ai/glm-5.2）／提供事業者ごとに入力・出力単価が異なるのがわかる。*

ざっくり比較すると、**「たくさん使う人ほど定額のCoding Planがお得、たまに使う・複数モデルを使い分けたい人はOpenRouterなどの従量課金が身軽」** という住み分けです。

## GLM-5.2の使い方①：Claude Codeにつないでコストを1/6にする

さて、Zenn 読者お待ちかねの本命。**Claude Code の頭脳を GLM-5.2 に差し替える** 方法です。使い勝手はそのまま、料金だけグッと下げられるのが狙いですね。

![Claude Code / Cline / Cursor を API 経由でモデルにつなぐ配線イメージ](https://raw.githubusercontent.com/Appare7/zenn-content/main/images/glm-5-2-tsukaikata/figure4.png)
*各ツールは API のエンドポイント（接続先URL）を差し替えるだけで、頭脳を GLM-5.2 に載せ替えられる。*

### 必要なもの

- Claude Code がインストール済みであること
- GLM Coding Plan（Liteでも可）または Z.ai の従量課金APIキー

Claude Code は Anthropic 形式のAPIをしゃべるので、GLM-5.2 側もそれに合わせた **コーディング用エンドポイント** を使います。

### 環境変数の設定

Claude Code は、接続先（ベースURL）と認証トークンを **環境変数** で差し替えられます。基本はこの2つ。

```bash
# GLM-5.2 のコーディング用エンドポイントに向ける
export ANTHROPIC_BASE_URL="https://api.z.ai/api/coding/paas/v4"
# Z.ai で発行したAPIキー（またはCoding Planのトークン）
export ANTHROPIC_AUTH_TOKEN="<あなたのZ.aiのキー>"
```

モデルの指定では、**1M（100万トークン）版を使うなら `glm-5.2[1m]`** というモデルIDを使います。末尾の `[1m]` を付けないと標準のコンテキスト長になる、と覚えておけばOK。Claude Code は内部で複数のモデル層（軽い処理用・重い処理用）を使い分けるので、それらをまとめて `glm-5.2[1m]` に向ける設定にしておくと、全部が1M版にルーティングされます。

### 1Mコンテキストをフルに使う設定

100万トークンの長文をちゃんと活かすには、あと2つ設定を足しておくと安心です。

```bash
# コンテキストを自動圧縮する前のウィンドウを1Mトークンに
export CLAUDE_CODE_AUTO_COMPACT_WINDOW=1000000
# 長いタスクでタイムアウトしないよう余裕をもたせる（約50分）
export API_TIMEOUT_MS=3000000
```

`CLAUDE_CODE_AUTO_COMPACT_WINDOW` は、文脈が長くなると自動で要約・圧縮する機能の発動ラインを1Mまで引き上げる設定。`API_TIMEOUT_MS` はタイムアウトの上限で、**1Mトークンもの巨大な文脈を扱うと最初の応答までに時間がかかる** ため、3,000,000ミリ秒（＝約50分）くらい余裕をもたせておくのがコツです。ここをケチると、長いタスクの途中で「時間切れ」になって悲しい思いをします。

設定したら、いつもどおり Claude Code を起動するだけ。見た目も操作も今までと同じまま、裏側だけ GLM-5.2 で動いている状態になります。うまくいけば、**月々の請求がだいぶ軽くなっているはず** です。

## GLM-5.2の使い方②：Cline / Cursor（OpenAI互換）

Claude Code 以外の人気ツール、**Cline** と **Cursor** でも GLM-5.2 は使えます。こちらは Claude 形式ではなく **OpenAI互換** のエンドポイントを使うのがポイント。設定先のURLとモデルIDが少し変わります。

- **ベースURL**：`https://api.z.ai/api/paas/v4/`
- **モデルID**：`glm-5.2`（Claude Codeの `[1m]` サフィックスは不要）
- **コンテキスト長**：`1000000`（100万）に設定して長文の恩恵を受ける

### Cline の場合

VS Code 拡張の Cline なら、設定画面でAPIプロバイダを「OpenAI互換（OpenAI Compatible）」にして、上のベースURLとAPIキー、モデル `glm-5.2` を入れるだけ。あとはいつもどおりタスクを投げれば、計画→ファイル編集→コマンド実行、と自律的に動いてくれます。コンテキストを100万に設定しておくと、差分やテスト出力まで全部視界に入れて作業してくれるのが気持ちいいところ。

### Cursor の場合

Cursor も同様に、OpenAI互換のカスタムモデルとしてベースURLとキー、モデルIDを登録すれば連携できます。エディタに統合された補完・チャットの頭脳を GLM-5.2 にできるわけですね。

:::message
**なぜツールごとに書き方が違うの？**
Claude Code は Anthropic（Claude）形式のAPIを前提にしているので `glm-5.2[1m]` とコーディング用URL、Cline / Cursor は OpenAI 形式なので `glm-5.2` と OpenAI互換URL——という違いです。中身のモデルは同じでも、しゃべる「プロトコル（通信の作法）」が違うだけ、とイメージするとスッキリします。
:::

## GLM-5.2の使い方③：APIを直接叩く／OpenRouter／セルフホスト

エディタ連携ではなく、**自作アプリやスクリプトから直接** GLM-5.2 を呼びたい人向けの選択肢もあります。

- **Z.ai の標準API**：OpenAI互換の関数呼び出し（function calling）形式に対応。既存の OpenAI SDK のベースURLを差し替えるだけで動かせるケースが多い。
- **OpenRouter 経由**：OpenRouter でAPIキーを発行し、モデルに `z-ai/glm-5.2` を指定すればOK。複数モデルを1つの窓口で管理できるのがラク。
- **セルフホスト**：MITライセンスなので、Hugging Face から重みを落として自前のGPU環境で動かすことも可能。ただし753BのMoEを快適に動かすにはそれなりのマシンが要るので、個人なら最初はAPI利用が現実的です。

エージェント用途では **OpenAI互換の関数呼び出しに対応** しているのが地味に効いていて、既存のエージェント基盤（Claude Code、Cline、Cursor など）に「エンドポイントとモデルIDを差し替えるだけ」でスッと組み込めるように設計されています。

## GLM-5.2を使う前に知っておきたい注意点

いいことばかり書いてきましたが、フェアに注意点も。

- **タイムアウト設定は必須級**：1Mコンテキストで長期タスクを回すと、最初の応答まで待たされます。前述の `API_TIMEOUT_MS` を長めに取っておかないと、途中で切れて泣きを見ます。
- **データの扱いは事前に確認**：業務のソースコードを外部APIに送ることになるので、機密情報を扱う場合は各プランの利用規約やデータの取り扱いポリシーを必ずチェックしましょう。気になるならセルフホストという逃げ道もあります。
- **ベンチ＝実力の保証ではない**：数字上は Claude Opus 4.8 に肉薄していても、あなたの実際のタスクで同じ体感になるとは限りません。**まずは安いLiteプランやOpenRouterの従量課金で小さく試す** のが賢いです。
- **価格だけで乗り換えない**：安さは魅力ですが、既存のワークフローやツールとの相性も大事。フロンティアモデルと **併用して、タスクによって使い分ける** のが現実的な落としどころです。

## まとめ

最後にギュッとまとめます。

- **GLM-5.2** は Z.ai が2026年6月中旬に公開した、コーディング特化のオープンモデル（MITライセンス・753BのMoE・100万トークン対応）
- コーディングベンチ **FrontierSWE で74.4%** と、GPT-5.5 を上回り Claude Opus 4.8 に肉薄
- 料金は **月18ドルの定額（GLM Coding Plan）** から。従量課金やOpenRouter経由も選べる
- **Claude Code なら、ベースURLとモデルID（`glm-5.2[1m]`）を差し替えるだけ** で頭脳を載せ替えられ、コストを大きく下げられる
- Cline / Cursor は OpenAI互換エンドポイント（`glm-5.2`）でつなぐ
- まずは安いプランで小さく試し、フロンティアモデルと使い分けるのがおすすめ

「AIコーディングの料金、そろそろ本気で見直したいな」という人にとって、GLM-5.2 は試してみる価値が十分にある選択肢です。使い慣れたツールはそのまま、頭脳だけこっそり載せ替えて、コスパを一段上げてみてくださいね。
