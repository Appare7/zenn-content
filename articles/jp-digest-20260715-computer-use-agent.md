ここ最近、「AIが自分でパソコンを操作する」系の話題が、ぐっと増えてきましたよね。ブラウザを開いて、フォームに入力して、ファイルを探して、必要ならターミナルまで触る。少し前まではデモっぽかったものが、かなり現実の作業に近づいてきた感じがあります。

そこに、2026年7月9日にMetaが **Muse Spark 1.1** を発表して、この分野がまた一段にぎやかになりました。いわゆる Computer Use（コンピュータ操作）や、コンピュータ操作エージェントの領域です。

この記事は、複数の海外記事や公式発表を突き合わせて、日本の読者向けに再構成したものです。1本の記事の翻訳ではなく、Claude・OpenAI・Gemini、そして新顔のMuse Spark 1.1を横並びで見ながら、「何が違うの？」「どれを使えばいいの？」を噛み砕いていきます。

## そもそも「Computer Use」って何？

![見て→考えて→クリックを繰り返す仕組み](/images/jp-digest-20260715-computer-use-agent/figure2.png)

Computer Use（コンピュータ操作）というのは、ざっくり言うと **AIが画面を見て、自分で次の操作を決めて、マウスやキーボードで実際に動かす仕組み** です。

流れとしてはこんな感じです。

AIがまず画面のスクリーンショットを見ます。そこで「今、どんな画面が表示されているか」を把握します。次に「じゃあ、このボタンを押そう」「この入力欄に文字を入れよう」と考えます。そして実際にクリックしたり、キーボード入力したりします。

操作が終わったら、また画面を見ます。想定とズレていたら、次の一手を修正します。つまり、**見る → 考える → 操作する** をぐるぐる回すわけです。人間がパソコンを使うときとかなり近いですよね。

ここで大事なのは、専用のAPI（外部から操作するための窓口）がないアプリでも動かせる可能性があるところです。従来の自動化だと、「このボタンはこのID」「この処理はこのAPI」みたいに、かなり決め打ちで作ることが多かったんです。

でもComputer Useは、人間と同じように画面を見て触ります。だから、APIが用意されていないアプリや、昔ながらのデスクトップアプリにも手を伸ばせる。ここが従来のスクリプト自動化とかなり違うポイントなんです。

もちろん、画面を見間違えたり、ボタンの位置が変わって迷ったりもします。万能の魔法ではありません。ただ、「AIに文章を書かせる」から一歩進んで、「AIに実際の作業を進めてもらう」方向に進んできた、というのが今の面白いところです。

## 今の主役は、ざっくり4強

![PCを操作するAI4強の比較](/images/jp-digest-20260715-computer-use-agent/figure3.png)

今この分野で名前がよく出てくるのは、ざっくり次の4つです。

まずは **Claude Computer Use**。画面を丸ごと操作できるタイプで、Windows / Mac / Linuxに対応しています。LinuxではVNC（遠隔デスクトップ接続の仕組み）経由でも動き、コンテナやVM（仮想マシン）でも使えます。ファイル操作やターミナル（コマンド入力の画面）まで扱えるので、かなり「普通にPCを触るAI」に近い立ち位置です。

特徴は、移植性（いろんな環境で動く度合い）が高いこと。環境を選びにくいので、デスクトップ全体をまたいだ作業に向いています。OSWorldという実務タスクのベンチマークでも首位級です。

次に **OpenAI**。以前の「Operator」は、今は **ChatGPT Agent** という形でブラウザ中心に動く方向です。さらに、2026年4月16日に **Codex の Background Computer Use** が登場しました。こちらは背景でmacのデスクトップを操作する方向に広がっていて、登場時点ではmacOSファーストです。開発作業を裏で進めてもらう、みたいな使い方を狙っている雰囲気ですね。

3つ目が **Gemini Computer Use**。こちらはブラウザ特化です。画面の構造データであるDOM（Document Object Model）を読んで動くので、速くて正確。特にフォーム入力のような、ブラウザ内で完結する作業に強いです。

そして新顔が、Metaの **Muse Spark 1.1**。2026年7月9日に登場しました。複数のサブエージェント（子分のAI）を並行で走らせるのが得意で、ツール使いの性能が高いモデルです。

こうして見ると、4者とも得意分野がけっこうバラバラなんです。「これ1つ入れれば全部OK」というより、**用途で選ぶ時代** になってきた、という感じですね。

## 新顔 Muse Spark 1.1 を、数字で見てみる

Muse Spark 1.1は、Metaが出したクローズドな多モーダルモデルです。クローズドというのは、モデルの重みが公開されていないという意味ですね。入力はテキスト・画像・音声に対応していて、出力はテキストのみです。

文脈は最大100万トークン。トークン（AIが文章を処理するときの単位）が大きいほど、長い会話や大きめの案件を覚えていられる量が増えます。100万トークンという数字は、長い作業の流れを扱うComputer Use系ではかなり重要なポイントです。

消費者向けには、Meta AIアプリの「Thinking」モードで無料で使えます。開発者向けには公開プレビューのAPIが用意されています。料金は、入力100万トークンあたり1.25ドル、出力100万トークンあたり4.25ドル。最初に20ドル分の無料クレジットも付いています。

一方で、オープンウェイト（重みの公開）はありません。つまり、手元にモデルを持ってきて再学習する、といった使い方はできません。このあたりは、使い方を考えるうえで押さえておきたいところです。

![Muse Spark 1.1とClaude Opus 4.8のベンチ比較](/images/jp-digest-20260715-computer-use-agent/figure4.png)

ベンチマークの数字を見ると、Muse Spark 1.1の性格がかなり見えてきます。比較対象としてClaude Opus 4.8がよく出てきます。

MCP Atlas（ツール使いの指標）では、Museが88.1、Opusが82.2です。ちなみにGPT-5.5は75.3なので、ここではMuseがかなりリードしています。

OSWorld-Verified（PC操作）では、Museが80.8、Opusが83.4。ここはわずかにOpusが上です。Terminal-Bench 2.1（ターミナル作業）でも、Museが80.0、Opusが82.7なので、ターミナル周りはOpusが少し強いですね。

SWE-Bench Pro（ソフト開発）では、Museが61.5、Opusが69.2。腰を据えたソフトウェア開発では、まだOpusに一歩譲る感じです。一方で、JobBench（実務のツール使い）では、Museが54.7、Opusが48.4。こちらはMuseが上回っています。

つまり読みとしては、**ツール連携や並行実行ではMuse Spark 1.1がかなり強い。ただし、じっくりコードを書いて直すような開発作業では、まだOpusが強い** という感じです。

開発ツール企業ClineのCEOであるSaoud Rizwan氏も、「Metaは本気でエージェント的コーディングを作っている。強いツール使いを、実際の大規模ワークロードを回せる価格で出してきた」と評価しています。

消費者目線の例も面白いです。ある海外記事では、夕食プランの例が紹介されていました。レストランを複数の情報源で比較して、営業時間を確認して、候補をまとめてくれる。最終判断は人間がする、という流れです。

ここで新しいのは、単に「おすすめを教えて」と聞くだけではないところです。AIが代わりに情報源を見に行き、比較して、手を動かしてくれる。まさにComputer Useらしい使い方なんです。

## 中身の話：ピクセルで見るか、HTMLで見るか

![ピクセル視覚型とDOM解析型の違い](/images/jp-digest-20260715-computer-use-agent/figure5.png)

Computer Useには、大きく2つの「画面の見方」があります。ここを押さえると、各ツールの違いがかなり分かりやすくなります。

1つ目は **ピクセル視覚型** です。これはスクリーンショットを撮って、画面上の座標をもとにクリックする方式です。人間が画面を見るのにかなり近いやり方ですね。

この方式の強みは、どんなアプリでも動かしやすいことです。ブラウザだけでなく、デスクトップアプリやファイル操作、ターミナルなども対象にできます。ClaudeやOpenAIのOperatorはこちら寄りです。

一方で、スクリーンショットを読んで判断するので、遅くなりやすく、トークンも食います。画面が広かったり、何度も確認が必要だったりすると、そのぶん処理量が増えるわけです。

2つ目は **DOM解析型** です。DOM（画面の構造データ）を読んで、ボタンや入力欄などの要素を直接指定する方式です。こちらはブラウザ限定ですが、速くて安くて正確です。Browser UseやGeminiはこちらのタイプです。

数字で見ると分かりやすいです。WebVoyager（実在サイトを巡回するベンチ）では、Browser Useが89.1%、OpenAI系のCUA（Computer-Using Agent、コンピュータ操作エージェント）が87%です。ブラウザ作業では、DOM解析型がかなり強いことが分かります。

一方で、OSWorld（デスクトップ操作のベンチ）でのClaude系は82〜83%台まで来ています。しかも初期のスコアは14.9%でした。ここ最近で一気に伸びた分野なんだな、と分かりますよね。

![出典元：Browser Use vs Operator vs Claude Computer Use](/images/jp-digest-20260715-computer-use-agent/source2.png)
出典: Particula Tech（https://particula.tech/blog/browser-use-vs-operator-vs-claude-computer-use-web-agents）

住み分けとしては、今のところかなりシンプルです。ブラウザだけで完結するなら、構造読みのDOM解析型。デスクトップ丸ごと触りたいなら、ピクセル視覚型。まずはこの目安で考えると選びやすいです。

## で、結局どう使い分ける？

ここまで見てくると、「で、自分はどれを使えばいいの？」となりますよね。複数の比較記事で共通していた見方をもとにすると、用途別の目安はこんな感じです。

ブラウザ自動化やフォーム入力なら、Geminiが向いています。DOMを読んで動くので、入力欄やボタンを正確に扱いやすく、ミスが少ないのが強みです。ブラウザ内で完結する事務作業や、フォームを順番に埋めていくような処理とは相性がいいです。

ファイル操作、PDFの読み込み、フォルダ整理、デスクトップ作業ならClaudeです。環境を選ばず動かしやすく、画面全体を見て操作できるので、「ブラウザだけでは終わらない作業」を任せやすいです。Windows / Mac / Linux、コンテナやVMまで含めて考えたいなら、かなり有力な選択肢になります。

Mac上での開発作業を背景で並行して動かしたいなら、OpenAIのCodexが候補になります。2026年4月16日に登場したBackground Computer Useは、登場時点ではmacOSファーストで、背景でmacのデスクトップを操作する方向へ広がっています。開発作業を裏側で進めたい人には気になるところですよね。

とにかく安く大量に回したいなら、Browser Useも候補です。自分でホストすれば、モデルのAPI代だけで済むからです。ブラウザ作業に絞れるなら、コスト面でかなり魅力があります。

大事なのは、「1つで全部やる」前提にしないことです。Computer Useはまだ発展中の領域なので、作業内容に合わせて選ぶのが今の正解に近いです。フォーム入力ならDOM解析型、デスクトップ作業ならピクセル視覚型、開発作業ならその環境に合うもの、というふうに分けて考えるのがよさそうです。

## 使う前に知っておきたい注意点

![大事な操作は人間が最後に承認する](/images/jp-digest-20260715-computer-use-agent/figure6.png)

ここまで読むと「もう全部AIに任せられるのでは？」と思いたくなるかもしれません。でも、ここは少し落ち着いて見ておきたいところです。

まず、ベンチマークのスコアは「保証」ではありません。むしろ「ここまでできる可能性がある」という天井に近いものです。実際の現場では、Cloudflare（bot対策）やCAPTCHA（画像認証）、レート制限、セッション切れなどで普通にコケます。

複数の記事でも、「スコアは床、つまり最低ラインだと思え。天井、つまりできる限界と勘違いするな」と釘を刺しています。ベンチで高いスコアが出ていても、自分の環境・自分のサイト・自分の作業で同じように動くとは限らない、ということですね。

だから、消す・買う・送るといった取り返しのつかない操作では、human-in-the-loop（人間が途中で承認する仕組み）を必ず挟むのが安全です。たとえば、最後の送信ボタンだけは人間が押す。購入や削除の直前で確認を求める。こういう設計にしておくと、事故のリスクをかなり下げられます。

現状ではまだ、「複数サイトを渡り歩けるか」「ページの見た目が変わっても対応できるか」「大きなコードベースを任せられるか」は未知数な部分も多いです。できることは増えていますが、過信は禁物です。

最初は、失敗しても困らない小さなタスクから試すのがいいです。たとえば情報収集、下書き、候補の整理、フォルダ内の確認、フォーム入力の補助あたりですね。慣れてきたら少しずつ任せる範囲を広げる。いきなり本番環境で全自動、みたいな使い方は避けたほうが安心です。

## まとめ

「AIが自分でPCを操作する」というテーマは、2026年で一気に実用ラインに近づいてきました。画面を見て、考えて、クリックする。人間がやっていた細かい作業に、AIがかなり踏み込んできています。

ただし、各モデルの得意分野ははっきり分かれています。Claudeは万能でデスクトップに強い。Geminiはブラウザで速い。OpenAI CodexはMac開発の背景作業に寄っている。Muse Spark 1.1はツール使いと並行実行が強い。そんな住み分けです。

なので、最初から「全部任せるAI」を探すより、「この作業ならどれが合う？」と考えるほうがうまくいきやすいです。ブラウザ作業なのか、デスクトップ作業なのか、開発作業なのか。そこを切り分けるだけでも、かなり選びやすくなります。

そして、消す・買う・送るような大事な操作は、必ず人間の承認を挟みましょう。Computer Useはかなり面白い領域ですが、まだ育ち盛りです。まずは小さいタスクから、承認つきで動かしてみてください。

気になるモデルから、ちょっと触ってみるくらいがちょうどいいと思います。たぶん、「あ、これは仕事の一部が変わるやつだ」と実感できるはずです。
