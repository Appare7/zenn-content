2026年、AIコーディングの合言葉がちょっと変わってきました。前は「Copilotか、Cursorか、Clineか。どれを選んで使い続ける？」という話が中心でしたよね。つまり発想は、ずっと「1体のAI・1つのワークフロー」だったわけです。

でも今は、「複数のエージェントを、どうケンカさせずに同時に動かす？」がテーマになってきています。1人の開発者が2〜3体のClaude Codeを走らせて、まるで小さな開発チームのマネージャーみたいに動く感じです。

この記事では、複数の海外の実践ガイドを読み比べて、サブエージェント（親から仕事を振られる子分のAI）とgit worktree（ギット・ワークツリー）を使った並列運用の型を、日本語で噛み砕いて再構成していきます。

ポイントは、「2体走らせても、待ち時間は1体とほぼ同じ。ひとつの待ち時間で“2つの意見”がもらえる」というところ。これ、うまく使えるとかなりおいしいんです。ただし、雑にやるとマージ地獄が待っています。そこも含めて見ていきましょう。

![Claude Codeを並列で動かす：サブエージェント×git worktree](/images/jp-digest-20260709-claude-code-parallel-agents/figure1.png)

## そもそも「並列で動かす」って何？サブエージェントとチーム、2つのやり方

AIエージェントを並列で動かす、と聞くと「ターミナルをいっぱい開いてClaude Codeを何体も起動するの？」と思うかもしれません。半分当たりですが、考え方としてはざっくり2種類あります。

ひとつ目は、サブエージェント方式です。

サブエージェント（親のAIから仕事を振られる子分のAI）は、いわば「分身」です。1つのセッションの中で、親のAIが仕事を切り分けて子分に投げ、結果をまた親のところに集めます。よくある形は、split-and-merge（分けて→まとめる）パターンです。

たとえば親が「この機能を作るには、認証まわり、UIまわり、テストまわりを見ないとね」と判断して、それぞれを子分に任せるイメージです。子分たちは親の文脈、つまり会話の流れを共有しています。なので「1人の頭の中で手分けしている」感じに近いです。

ふたつ目は、チーム方式です。

こちらは、それぞれのエージェントが独立して考えます。真ん中に共有のタスクリストを置いて、「誰が何をやっているか」を見える化します。そして着手前に「今からこれをやるよ」と旗を立てて、同じ作業に突っ込まないようにします。

さらに、gitの履歴を見れば、お互いの作業も観察できます。別々の脳を持った仲間が、タスクボードを見ながら連携している感じですね。サブエージェントが「1人の頭の中で手分け」なら、チーム方式は「別々の脳を持った仲間がタスクボードで連携」です。

どちらが正解というより、使いどころが違います。軽く分解できる作業ならサブエージェント、独立した検討や実装を同時に進めたいならチーム方式、という見方をするとわかりやすいですよ。

![「分身」と「チーム」——並列運用の2つの型](/images/jp-digest-20260709-claude-code-parallel-agents/figure2.png)

## 主役は git worktree（ギット・ワークツリー）

複数エージェントを本当に同時に走らせるなら、土台としてかなり重要になるのが git worktree（ギット・ワークツリー）です。

git worktreeとは、1つのリポジトリから複数のブランチを、別々のフォルダで同時にチェックアウトできる仕組みです。普通に考えると、「じゃあリポジトリを丸ごと何個もクローンすればいいのでは？」となりますよね。

でもworktreeは、丸ごとクローンとは違います。.git（履歴やオブジェクト）は1つを共有しつつ、作業ディレクトリだけを分けます。なのでディスクにも優しいんです。部屋は複数あるけど、倉庫は共通、みたいなイメージです。

基本コマンドは、まずこの3つを押さえれば大丈夫です。

```bash
git worktree add ../my-project-feature-a feature/add-auth
```

これは、新しい作業部屋を作るコマンドです。`../my-project-feature-a` という別フォルダに、`feature/add-auth` ブランチの作業場所を用意します。

```bash
git worktree list
```

これは、今ある作業部屋の一覧を見るコマンドです。

```bash
git worktree remove ../my-project-feature-a
```

これは、作業が終わった部屋を片づけるコマンドです。

![git worktreeは1つの.gitを共有して作業ディレクトリだけを分ける](/images/jp-digest-20260709-claude-code-parallel-agents/figure3.png)

ポイントは、「各エージェントに自分専用のブランチと自分専用のディレクトリを渡す」ことです。これで、ファイルの衝突なし、状態のにじみ出しなし、という状態を作れます。

同じフォルダで複数のAIに触らせると、片方が編集中のファイルをもう片方が上書きする、みたいな事故が起きます。人間同士でもイヤなのに、AI同士でやられると、もう机の上でケーブルが全部からまった状態です。worktreeは、それを避けるための「作業部屋分け」なんです。

![MindStudioの実践ガイド。git worktree・DB分岐・ポート隔離で複数のClaude Codeセッションを同時に走らせる方法を解説している](/images/jp-digest-20260709-claude-code-parallel-agents/source1.png)

## 実際の手順：エージェントごとに"作業部屋"をまるごと分ける

ただし、worktreeでファイルを分けるだけでは足りません。複数のAIエージェントを同時に走らせるなら、ファイルだけでなく、DBやポートまで分ける必要があります。

海外ガイドでよく挙げられている隔離の4点セットは、こんな感じです。

1つ目は、ファイル隔離です。エージェントごとにworktreeを作り、別ブランチをチェックアウトします。これはさっき見た通りですね。

2つ目は、データベース隔離です。各worktreeの `.env.local` に、それぞれ別の `DATABASE_URL` を設定します。これをやらないと、別々のエージェントが同じDBを触ってしまい、DBの混線や状態破壊が起きます。片方がテスト用に入れたデータを、もう片方が前提として使ってしまう、みたいなことですね。

3つ目は、ポート隔離です。それぞれに別の `PORT` を割り当てます。たとえば `3001`、`3002` のように分けます。同じポートで起動しようとして「もう使われています」と怒られるのは、並列以前のつまずきポイントです。

4つ目は、タスクの明確化です。各エージェントには、小さく、境界のはっきりした指示を渡します。「いい感じに全体を改善して」みたいな依頼は、並列運用では危険です。どこまで触っていいのかがぼやけると、他のエージェントの作業範囲と重なりやすくなります。

![エージェント隔離の4点セット：ブランチ／DATABASE_URL／PORT／小さく明確なタスク](/images/jp-digest-20260709-claude-code-parallel-agents/figure4.png)

マージのときも、いきなり合体させないのが大事です。まず差分を見ます。

```bash
git diff main...feature/add-auth
```

これで、`main` と `feature/add-auth` の差分を確認してから進めます。

また、土台になる変更、つまり複数ブランチが共通して必要とする部分は先にマージしておくと、衝突が減ります。並列作業では、衝突は「同時編集の最中」ではなく「マージのタイミング」でまとめて出ます。これは逆に言うと、扱いやすい形に先送りできるということでもあります。

もちろん、先送りしたぶん最後に雑に処理すると爆発します。なので、差分確認と順番づけがかなり大事なんです。

## 何体まで同時に動かせる？ 2〜5体の壁

では、AIエージェントは何体まで同時に動かせるのでしょうか。

アーキテクチャ上のハードな上限はありません。理屈の上では、もっとたくさん並べることもできます。でも現実的には、2〜5体くらいが扱いやすいレンジです。

複数のガイドで共通している見方は、それを超えるとコーディネーション（連携）のオーバーヘッドが、並列化のうまみを上回りやすい、というものです。要するに、AIを増やしすぎると「仕事を進める時間」より「誰が何をやったか整理する時間」が増えてしまうんですね。開発チームでも、人が増えれば自動的に速くなるわけではないのと同じです。

コスト面も見逃せません。3体を並列で走らせると、トークンの消費もざっくり3倍のスピードで進みます。なので走らせる前に、レート制限（使用量の上限）を意識しておくのが大事です。気づいたら使用量だけ元気に走っていた、なんてことになりかねません。

一方で、時間の面ではかなりおいしいです。2時間かかる直列タスクでも、うまく分けられれば「依存関係の一番長い一本道」の時間まで縮められます。

たとえば、Aが終わらないとBに進めない作業は短くできません。でも、AとBとCが互いに独立しているなら、同時に走らせられます。だから「1体分の待ち時間で複数の成果」が狙えるんです。

## いちばんの敵は「マージ地獄」：並列がコードを壊す理由

ここからが本題です。並列運用でいちばん怖いのは、速くなることではありません。速く壊れることです。

2026年の本番マルチエージェント運用で「単一で最大のボトルネック」と言われるのが、このコーディネーション（連携）問題です。制御なしで並列エージェントを走らせると、コードベースは普通に壊れます。

![並列がコードを壊す3つのパターン：ホットスポット衝突／二重実装／意味の矛盾](/images/jp-digest-20260709-claude-code-parallel-agents/figure5.png)

理由は主に3つあります。

1つ目は、同じ場所を取り合うことです。ルーティング、設定ファイル、登録ファイルのような“ホットスポット”に複数エージェントが同時に手を入れると、マージ衝突が起きます。みんなが同じ引き出しを同時に開けようとして、ガチャガチャになるやつです。

2つ目は、二重実装です。各エージェントは部分的で孤立した文脈で動くので、同じ機能を別々に実装してしまうことがあります。片方はAという形で、もう片方はBという形で、同じ目的のコードを書いてしまう。動くけれど、あとから見ると「なんで同じものが2個あるの？」となります。

3つ目が、いちばん厄介な「意味の矛盾」です。

これは、コンパイルもlint（静的チェック）も通るのに、実行すると噛み合わないタイプの問題です。たとえばタイムゾーンの扱いが食い違っていたり、業務ロジックの前提がずれていたりします。機械的なチェックでは通ってしまうので、発見が遅れがちです。

並列AI運用の問題をひと言でいうなら、こうです。AIエージェントは、素早く重なり合う変更を、部分的で孤立した文脈のまま生成します。だから、マージ衝突、重複実装、意味の矛盾が増えます。

つまり、AIが悪いというより、段取りなしに並列化するのが危ないんです。速い包丁を何本も同時に振り回したら、そりゃキッチンは大騒ぎになりますよね。

## 壊さないための型：隔離 → 検証ゲート → 順番マージ

では、どうすれば壊さずに並列化できるのでしょうか。

海外ガイドが共通で薦めている安全策は、かなりシンプルです。並列でやるのは“作業”だけ。統合、つまりマージは同時にやらない。

型としては、こうです。まず、worktreeで隔離して、各エージェントがバラバラに作業します。次に、マージ前に検証ゲートを通します。最後に、`main` には1本ずつ順番に統合します。

各ブランチは、`main` にリベースしてから入れると、想定外の衝突を「1度に1ブランチ分」に抑えられます。一気に全部混ぜるのではなく、1本ずつ確認しながら入れるわけです。地味ですが、これがかなり効きます。

![隔離→検証ゲート→順番マージ、という壊さないための型](/images/jp-digest-20260709-claude-code-parallel-agents/figure6.png)

検証ゲートに入れるものは、いくつかあります。

まず、自動テストです。単体テストと結合テストを通します。次に、静的解析やセキュリティスキャンです。lint（静的チェック）だけでなく、危ない変更がないかも見ます。

さらに、AIによるコードレビューも使えます。仕様と照らして、実装がズレていないかを見てもらいます。ただし、最後は人間のレビューが必要です。意味の正しさ、つまり業務ロジックやタイムゾーンの扱いのような部分は、人が見るべきところです。

作業の割り振り方にも型があります。

互いに依存しない作業、たとえば調査、監査、別案出しのようなものは、ファンアウト（ばらまいて並列）に向いています。複数のエージェントに同時に投げて、それぞれの結果を見比べる形です。

一方で、順番に依存する作業、たとえば調査→執筆→レビューのようなものは、パイプライン（順番に流す）に向いています。前工程の結果を次に渡していく形ですね。

高リスクな変更では、クロスチェックも有効です。セキュリティ、データ移行、性能に効くコードのような変更では、2体に同じ課題を独立して解かせて、出力を見比べてから採用します。

ただし、トークン代はざっくり倍になります。そして大事な注意があります。2体の意見が一致しても、それが正しいとは限りません。あくまで1つのシグナルであって、専門家レビューの代わりにはなりません。

ここを勘違いすると、「AIが2体とも同じことを言ってるから正解！」となりがちですが、2人そろって同じ勘違いをすることもあります。人間でもありますよね、会議で全員うなずいたのに後で間違いに気づくやつです。

## で、結局どう使い分ける？（現実的な落としどころ）

ここで、少し現実チェックもしておきましょう。

![Scopirの解説記事。Claude・Codex・Copilotを並列で走らせるオーケストレーションの型を、実践寄りに整理している](/images/jp-digest-20260709-claude-code-parallel-agents/source2.png)
*出典: Scopir（https://scopir.com/posts/multi-agent-orchestration-parallel-coding-2026/）*

AIエージェントは、1つの課題を解くタイプのベンチマークであるSWE-Bench Verified（ソフトウェア修正能力を見る評価）では、トップモデルが70%超です。かなり強いですよね。

でも、複数ファイル、4ファイル以上、平均107行にまたがる変更を扱うSWE-Bench Pro（より広く散らばった変更を見る評価）では、最良のモデルでも25%を下回ります。

つまり、AIは「1つのまとまった課題」にはかなり強くなっている一方で、「大きく散らばった変更」はまだ苦手なんです。だからこそ、人間が作業の切り分けと検証を握るのが肝心です。

![Augment Codeのガイド。隔離→検証ゲート→順番マージという安全な統合の型を6つのパターンで解説している](/images/jp-digest-20260709-claude-code-parallel-agents/source3.png)
*出典: Augment Code（https://www.augmentcode.com/guides/how-to-run-a-multi-agent-coding-workspace）*

ソロ開発者なら、まずは1つの機能をworktreeで2体に分けてみるのがおすすめです。いきなり5体を並べる必要はありません。うまく境界を切れる小さめのタスクから始めてみてください。

たとえば、片方には実装、もう片方にはテストやレビュー観点の洗い出し、というように分けると始めやすいです。大事なのは、「どこまで触っていいか」がはっきりしていることです。

小さめのチームなら、タスクの種類ごとにエージェントの担当を決めるとよさそうです。さらに、リポジトリに共有設定を置いて、どのMCP（外部ツールなどとつなぐための共通口）を使うか等をそろえると、チーム内で運用がブレにくくなります。

結局のところ、AIを“何体”並べるかより、“どう切り分けて・どう検証して・どう順番にマージするか”が勝負です。

並列は魔法ではありません。段取りです。段取りがあるから速くなるし、段取りがないと速く壊れます。

## まとめ

- サブエージェントは「分身」、チーム方式は「独立した仲間」。並列には大きく2つのやり方があります。
- 土台は git worktree。1つの `.git` を共有して、作業部屋だけを分けます。DBとポートも忘れずに分けましょう。
- 現実的に扱いやすいレンジは2〜5体。3体で走らせると、消費トークンもざっくり約3倍のスピードで進みます。
- 最大の敵はマージ地獄です。衝突、二重実装、意味の矛盾を防ぐには、隔離→検証ゲート→順番マージの型が効きます。
- 大きく散らばった変更は、AIがまだ苦手です。作業の切り分けと検証は、人間が握る必要があります。

まずは、小さいタスクをworktreeで2体に分けるところから試してみてください。うまくハマると、「AIを1体に順番待ちさせていた時間」が、ちょっともったいなく感じてくるはずです。
