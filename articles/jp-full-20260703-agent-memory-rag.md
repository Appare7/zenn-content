AIエージェント（自律的にタスクをこなすLLMの仕組み）に「記憶（メモリ）」を持たせるとき、多くのシステムは **RAG（検索拡張生成 / Retrieval-Augmented Generation）** のパイプラインをそのまま使っています。でも、RAGがもともと想定していた状況と、エージェントの記憶が置かれている状況って、じつはかなり違うんです。この論文は「その前提のズレ」を出発点に、記憶検索を根っこから作り直す提案をしています。気軽に読んでいきましょう。

## 要約（Abstract）

エージェントのメモリシステムは標準的なRAGパイプラインを採用しがちですが、この設定では前提となる仮定が異なります。RAGが対象とするのは大規模で多様な文書群（コーパス）で、取り出される文章は互いにバラバラです。一方でエージェントの記憶は、**閉じていて一貫した対話の流れ（stream）** であり、そこに含まれる区間（span）は互いに強く関連していて、しばしば重複しています。

この違いのもとでは、固定の上位k件（top-k）類似度検索は冗長な文脈を返しがちで、事後の枝刈り（pruning、剪定）は、正しい推論に必要な「時間的につながった前提」まで削ってしまいます。そこで本論文は、検索は類似度マッチングを **超える（beyond）** べきだと主張します。すなわち、記憶を潜在的な部品（latent components）に **ほどいてから束ねる（decoupling to aggregation）**——記憶を意味的な部品に分解し、それらを階層に組み上げ、その構造を使って検索を駆動する、という考え方です。

提案手法 **xMemory** は、「そのまま保てる単位（intact units）の階層」を構築し、**スパース性と意味的忠実性（sparsity–semantics）** の目的関数によって記憶の分割・統合を導きながら、検索可能でありつつ内容に忠実な高レベルのノード構成を維持します。推論時、xMemory は上から下へ（top-down）検索し、複数の事実にまたがる質問に対しては、テーマや意味のコンパクトで多様な集合を選び、読み手の不確実性（uncertainty）が減るときにだけエピソードや生のメッセージへと展開します。**LoCoMo** と **PerLTQA** のベンチマークで、回答品質とトークン効率の両面で一貫した改善が確認されました。

## はじめに（Introduction）

![RAGの前提とエージェント記憶の前提の違い](https://raw.githubusercontent.com/Appare7/zenn-content/main/images/jp-full-20260703-agent-memory-rag/figure2.png)

RAGはもともと、**大規模で多様なコーパス** のために設計されました。取り出される文章は多様で、いちばんの失敗パターンは「無関係なものを引いてしまうこと（irrelevance）」です。ところがエージェントの記憶は、**閉じた一貫した流れ** を作ります。区間どうしは強く相関していて、しばしばほぼ重複（near duplicates）しているんですね。

この違いのせいで、標準RAGをエージェントの記憶にそのまま当てると、次のような困りごとが起きます。固定の上位k件類似度検索は、**一つの密集した領域に潰れて（collapse）** しまい、冗長な証拠を返して、「本当に必要なもの」と「ただ似ているだけのもの」を切り分けられません。かといって、事後に枝刈りをすると事態は悪化します。というのも、会話中の証拠は **共参照（co-reference、同じ対象を指す言い換え）** や **省略（ellipsis）**、タイムライン上の依存関係を通じて時間的に絡み合っているからです。似ているからと削った区間が、じつは後の推論に不可欠な前提だった——ということが起こります。

![標準RAGの2つの落とし穴：冗長と前提の消失](https://raw.githubusercontent.com/Appare7/zenn-content/main/images/jp-full-20260703-agent-memory-rag/figure3.png)

さらに、重複を避けようとして「要約中心の階層（summary-centric hierarchies）」を作ると、今度は候補どうしを区別するための **微妙な差（subtle details）** がぼやけてしまいます。要約しすぎると、似た履歴の中で「これとあれは何が違うのか」を判別できなくなるわけです。

そこで本論文が提案する解決の筋道が **「束ねる前に、まずほどく（decoupling before aggregation）」** です。流れを意味的な部品に分解し、それらを階層に組み立て、そのうえで構造を逆にたどって検索を駆動します。検索は「生の区間に対する類似度ランキング」ではなく、「分解と集約が生み出した構成（organisation）」によって決まるべきだ、という主張です。

これを実装したのが **xMemory** で、記憶構造の管理（memory structure management）と適応検索（adaptive retrieval）を組み合わせています。生のメッセージから抽象的な単位までの **4層の階層** を構築し、その階層に対して上から下へ検索します。

> (訳注: ざっくり言うと「似た記憶をそのまま並べて上からk件取る」のをやめて、「記憶を事実の粒にほどいて分類しておき、必要なテーマから降りていく」というアプローチです。)

## 関連研究（Related Work）

論文では大きく2つの流れが整理されています。ひとつは **LLMエージェントの記憶管理（Memory Management in LLM Agents）** で、対話履歴をどう保存・要約・更新していくかという研究群。もうひとつは **エージェント記憶における検索と文脈管理（Retrieval and Context Management）** で、必要な情報をどう取り出し、限られた文脈長にどう収めるかという研究群です。xMemory は、この両者を「構造」と「検索」の両輪として統合的に扱う点に特徴があります。

## 手法（Methodology）

### 問題設定と全体像

扱う問題は、長い対話の記憶から、質問に答えるために必要な文脈を過不足なく取り出すことです。xMemory は2つの要素を密に結びつけます。ひとつは記憶の **構造管理**（生のメッセージから抽象的な単位までの4層階層を、スパース性と意味の指針で組み上げる）。もうひとつは **上から下へたどる検索**（関連するテーマ・意味をまず選び、不確実性の低減に応じてエピソードやメッセージへ展開する）です。

### 記憶構造の管理（Memory Structure Management）

![xMemoryの4層階層：メッセージ→エピソード→意味→テーマ](https://raw.githubusercontent.com/Appare7/zenn-content/main/images/jp-full-20260703-agent-memory-rag/figure4.png)

**4層の階層** は次のように対応づけられます。

- **メッセージ（Messages）**: 生の発話そのもの。
- **エピソード（Episodes）**: 連続したメッセージのまとまり（局所的な出来事）。各メッセージ・ブロックはちょうど1つのエピソードに対応します。
- **意味 / 事実（Semantics）**: エピソードから抽出される、再利用可能な事実の粒。1つのエピソードから複数の意味ノードが生まれます。各意味ノードはちょうど1つのテーマに割り当てられます。
- **テーマ（Themes）**: 関連する意味ノードを束ねた高レベルの抽象単位。1つのテーマは複数の意味ノードを含みます。

この階層を良い形に保つために、**スパース性と意味の指針関数** `f(𝒫) = SparsityScore(𝒫) + SemScore(𝒫)` を使います。ざっくり2つの狙いのバランスを取るものです。

- **SparsityScore（スパース性スコア）**: テーマのサイズが偏らないようにするための項。テーマの大きさが均等に近いほどスコアが上がり、**密集領域への潰れ（dense region collapse）** を抑えて、高レベル検索の効率を上げます。
- **SemScore（意味スコア）**: テーマ内の意味的なまとまりを保ちつつ、孤立したテーマ（論文いわく "semantic islands"、意味の孤島）ができるのを避ける項。意味埋め込みとテーマ重心（centroid）とのコサイン類似度の平均などで測ります。

新しい意味ノードは、重心との類似度がもっとも高いテーマに **アタッチ（attach）** されます。混み合いすぎたテーマはクラスタリングで **分割（split）** され、小さすぎるテーマは近いテーマと **統合（merge）** されます。分割・統合の判断は、上記の `f(𝒫)` を最大化するように行われます。加えて、テーマや意味のノード間に疎な **k近傍グラフ（kNN graph）** を張り、全体を総当たりで維持することなく効率的にたどれるようにしています。

> (訳注: 「事実の粒（意味ノード）をテーマという棚に仕分けし、棚が偏らない・意味が散らばりすぎないように、棚の分割と統合を自動でやる」というイメージです。)

### 適応検索（Adaptive Retrieval）

![適応検索の2段階：代表選択と不確実性適応](https://raw.githubusercontent.com/Appare7/zenn-content/main/images/jp-full-20260703-agent-memory-rag/figure5.png)

検索は2段階です。

**Stage I：質問に応じた代表選択（Query-aware representative selection）**
まず質問（クエリ）に対して、埋め込みベースの検索で候補となる意味ノードとテーマを絞り込みます。そのうえで、k近傍グラフの上で **「網羅の増分（coverage gain）」** と **「質問との関連（query relevance）」** を両立するよう、貪欲法（greedy）で代表を選んでいきます。各ステップでは、まだカバーできていないノードを新しく覆えるほど、かつ質問に関連するほど高評価になるノードを選びます。まずテーマを選び、そこから導かれる意味を選ぶことで、集約と多段推論（multi-hop reasoning）を支える **コンパクトな骨格（backbone）** を作ります。

**Stage II：不確実性に応じた証拠の追加（Uncertainty adaptive evidence inclusion）**
次に、選ばれた骨格に紐づくエピソードを「そのまま保てる単位」として集め、質問との類似度と意味的な裏づけで順位づけします。そして、そのエピソードが読み手の **予測の不確実性を十分に下げるとき** だけ、証拠として取り込みます。生のメッセージへの展開も、不確実性の低減が「追加するトークンに見合う」ときだけ行い、追加してももう確信度が上がらなくなったら早めに打ち切ります（early stopping）。

つまり「似ている順に上からk件」ではなく、「まず多様で代表的な骨格を選び、迷いが減るぶんだけ細部に降りる」という設計です。これによって、**短くても十分な文脈** を作れるわけですね。

## 実験（Experiments）

### 設定

**評価データセット** は、長距離の会話推論を測る **LoCoMo** と、個人の長期記憶に関する質問応答 **PerLTQA** の2つ。**評価指標** には BLEU / F1 / ROUGE-L といった回答品質の指標に加えて、効率を測る **1クエリあたりのトークン数（tokens/query）** を使います。**バックボーンLLM** は Qwen3-8B、Llama-3.1-8B-Instruct、GPT-5 nano の3種類。**比較対象（ベースライン）** は Naive RAG（素朴なRAG）、A-Mem、MemoryOS、LightMem、Nemori などです。

### 主な結果

![ベンチマーク結果：質は上、トークンは約半分](https://raw.githubusercontent.com/Appare7/zenn-content/main/images/jp-full-20260703-agent-memory-rag/figure6.png)

**LoCoMo** では、xMemory が3つのバックボーンすべてで平均性能トップでした。Qwen3-8B の場合、平均 BLEU は 34.48（Nemori の 28.51 に対して）、F1 は 43.98（同 40.45）で、1クエリあたりのトークン数は 4,711（A-Mem の 9,103 に対して）と大幅に少なくなっています。GPT-5 nano では平均 BLEU 38.71 / F1 50.00 を、トークン数 6,581（Nemori は 9,155）で達成。改善は特に **長距離の推論** で顕著で、たとえば時間的な（temporal）設問では、Qwen3-8B で BLEU 23.60・F1 33.74 から、29.58・37.46 へと伸びています。

**PerLTQA** でも同様に良好で、Qwen3-8B で BLEU 36.24 / F1 47.08 / ROUGE-L 42.50 を、トークン数 5,087（MemoryOS は 6,499）で達成しました。個人の長期記憶という別種のタスクへもうまく一般化しています。

**効率** の面では、証拠の網羅（full answer coverage）に必要な平均ブロック数が xMemory で 5.66 と、Naive RAG の 10.81、RAG+枝刈りの 13.31 よりずっと少なく、トークンにして 974.56（それぞれ 1,979.26、1,587.99）でした。しかも品質は BLEU 34.48 / F1 43.98（Naive RAG は 27.92 / 36.42）と上。**同じ正解にたどり着くのに、約半分の文脈で済む** ということです。

## 分析（Analysis）

### アブレーション（要素を外す実験）

LoCoMo（Qwen3-8B）で各要素の寄与を調べたところ、**階層構造だけ（memory-only）** でもすでに Naive RAG を上回りました。そこに **代表選択（+RepSel）** を足すとさらに向上し、**不確実性に応じた追加（+UncSion）** を足すと低レベルの展開が洗練され、フル構成で最高精度・最小トークンに達しました。構造と検索の両方が効いていることが確認できます。

### RAGにおける証拠密度と枝刈りの影響

Naive RAG、RAG+枝刈り、xMemory で、取り出したブロックあたりの「当たり（hit）」の分布を比べています。xMemory は **2件当たり・複数件当たりの割合が高く**、1ブロックあたりの証拠密度が高いことを示します。逆に枝刈りは分布を「1件当たり」に寄せてしまい、密度がむしろ下がりました。冗長を削るはずの枝刈りが、証拠のまとまりを壊しているわけです。

### 記憶構築における遡及的な再構成

分割・統合の操作を有効にすると、ノードの動的な再割り当て率が **44.91%**（凍結時は 0%）に上がり、F1 も改善しました。記憶の構造を後からしなやかに組み替えられること（structural plasticity）が、下流のQA性能を支えているという結果です。

## 結論（Conclusion）

本論文は、エージェント記憶の検索は **「区間のマッチングを超えて、一貫した流れの潜在的な部品を浮かび上がらせるべき」** だと主張しました。xMemory は、記憶構造の管理と適応検索を組み合わせ、**冗長を抑えつつ証拠の完全性（integrity）を保ちます**。LoCoMo と PerLTQA において、多様なオープンソース／クローズドソースのLLMバックボーンで、回答品質とトークン効率の一貫した改善を示し、証拠密度と網羅の分析が「標準RAGの検索と枝刈りがなぜエージェント記憶で限界を迎えるのか」を裏づけました。

鍵となる発想は、記憶を階層に組んで **強く相関した記憶を意味の部品にほどく** こと。これによって、**冗長を減らしながら、時間的につながった証拠を保つ** 検索が可能になります。RAGをそのまま流用するのではなく、エージェント記憶ならではの前提に合わせて検索を設計し直す——そんな一歩を示した研究でした。
