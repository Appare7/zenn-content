![AIエージェントの作り方2026、フレームワークをどう選ぶかのアイキャッチ](/images/jp-digest-20260712-ai-agent-framework/figure1.png)

2026年にAIエージェント（人間の代わりに考えてツールを使い、タスクをこなすAI）を「よし、自分でも作ってみるか」と思ったとき、最初に詰まりがちなのって、実はコードそのものじゃないんですよね。

いきなり出てくるのが、「で、どのフレームワークを使うの？」問題です。

LangGraph、CrewAI、AutoGen、smolagents、OpenAI Agents SDK、Google ADK、Mastra、Dify……名前だけでちょっとお腹いっぱいです。ラーメン屋の券売機でメニューが多すぎて後ろの人にプレッシャーを感じる、あの感じに近いです。

ただ、2026年は少し流れが変わってきました。フレームワークが雨後の筍みたいに増えた時期を過ぎて、「乱立」から「淘汰・統合」へ動き始めた年なんです。だからこそ、今いったん見取り図を作っておくと、かなり選びやすくなります。

この記事は、複数の海外記事と公式発表をもとに、日本のエンジニア向けに再構成したオリジナル解説です。丸写しや翻訳ではなく、「結局どう選べばいいの？」に寄せて見ていきます。

## そもそもフレームワークって何をしてくれるの？

![フレームワークが肩代わりする4つ（ループ管理・状態記憶・ツール呼び出し・マルチエージェント協調）を並べた図](/images/jp-digest-20260712-ai-agent-framework/figure2.png)

AIエージェントの中身をかなりざっくり言うと、こんなループです。

LLM（大規模言語モデル）が考える。  
必要ならツールを使う。  
ツールの結果を見て、また考える。  
まだ足りなければ、もう一回ツールを使う。

この「考える→使う→見る→また考える」のループを毎回ゼロから書くのは、けっこう大変です。最初は楽しくても、状態管理やエラー処理が増えてくると、だんだん“手作り感あふれる迷路”になりがちなんですよね。

そこでフレームワークの出番です。主に肩代わりしてくれるのは、次の4つです。

1つ目は、エージェントのループ管理。  
LLMに何を渡して、どのタイミングでツールを呼び、結果をどう次の判断につなげるか、という流れを扱ってくれます。

2つ目は、状態・記憶の管理。  
会話履歴や途中経過、「いま何をやっている途中か」といった情報を保持します。人間でいう作業メモですね。これがないと、エージェントが数分前の自分を忘れる鳥頭ムーブをします。

3つ目は、ツール呼び出し。  
検索、DB操作、ファイル操作、社内APIなど、外部の道具をAIに使わせる部分です。

4つ目は、マルチエージェントの協調。  
複数のエージェントに「調査担当」「実装担当」「レビュー担当」みたいな役割を持たせて、チームとして動かす仕組みです。

つまりAIエージェントの「作り方」を考えるというのは、この4つをどのフレームワークに、どこまで任せるかを選ぶ話なんです。

## 2026年の勢力図

![2026年の主要AIエージェントフレームワークを役割で並べた勢力図](/images/jp-digest-20260712-ai-agent-framework/figure3.png)

では、2026年時点でよく名前が出てくるフレームワークを、役割ベースで見ていきます。

まずLangGraph。  
状態をグラフで管理するタイプで、本番・企業向けの本命としてよく扱われています。エージェントの処理を「どこからどこへ進むか」「途中で何を保持するか」といった形で設計しやすいのが特徴です。

CrewAIは、役割を持つエージェントの「クルー」、つまりチームを少ないコードで組めるのが売りです。手早く試すにはかなり向いています。社内向けの作業自動化や、コンテンツ生成の流れ作業などでは、とっつきやすい選択肢です。

AutoGenは、会話ベースのマルチエージェントで先行していた存在です。ただし後で触れるように、今は単体では保守モードという扱いになっていて、新しく始めるなら統合版を見る流れになっています。

smolagentsは、LLMにPythonコードを書かせて動かす「コードファースト」派です。Python（データ処理や自動化でよく使われるプログラミング言語）でデータ処理やスクリプトを回したい人には、かなり気になる存在です。

ほかにも、OpenAI Agents SDK（軽量な試作向き）、Google ADK（Google Cloud向き）、Mastra（TypeScript派の本命）、Dify（ノーコード寄りの画面で組める）などがひしめいています。

Mastraについては、ReplitのAgent 3に採用され、タスク成功率が80%から96%に上がった例や、ある企業が7万5000人規模に配った例も紹介されています。TypeScript（JavaScriptに型を足した言語）で固めたいチームにとっては、かなり現実味のある候補です。

ここで1つ注意です。人気の指標としてGitHubのスター数を見ることがありますが、媒体によって数え方や時点が違って、数値がけっこう割れます。

たとえば、ある比較ではAutoGenが約4.2万、CrewAIが約3.1万スターと紹介されています。一方、別の集計ではLangGraphやCrewAIが5万超と出ることもあります。どれも数万スター級で拮抗している、くらいに見ておくのがよさそうです。

スター数だけで「これが最強！」と決めるのは、レビュー数だけで居酒屋を決めるくらい危ういです。もちろん参考にはなります。でも、自分の用途に合うかは別の話なんですよね。

## いちばんの事件：AutoGenとSemantic Kernelが「合体」した

![AutoGenとSemantic KernelがMicrosoft Agent Framework 1.0に統合されたことを示す図](/images/jp-digest-20260712-ai-agent-framework/figure4.png)

2026年の大きな動きとして外せないのが、AutoGenとSemantic Kernelの統合です。

2026年4月3日、これまで別々だった2つの土台「AutoGen」と「Semantic Kernel」を1つに統合した、Microsoft Agent Framework 1.0が本番リリースされました。.NET（主に業務アプリ開発で使われる開発基盤）とPythonの両方で使えます。

![Microsoft Agent Framework 1.0の統合を報じるSean Kimの記事](/images/jp-digest-20260712-ai-agent-framework/source2.png)
*出典: Sean Kim — Arts & Tech（https://blog.imseankim.com/microsoft-agent-framework-1-0-semantic-kernel-autogen-dotnet-python/）*

これはかなり象徴的です。まさに「フレームワーク乱立」から「統合」へ向かう流れが、はっきり見えた出来事でした。

中身で注目したいのは、MCP（モデルに外部ツールをつなぐ共通規格）とA2A（エージェント同士をつなぐ連携規格）への対応です。

AIエージェントを作っていると、「このモデルからこのツールを呼びたい」「このエージェントと別のエージェントを連携させたい」という話がすぐ出てきます。そこで規格がバラバラだと、つなぐだけで疲れます。家中の充電ケーブルが全部違う端子、みたいな地獄です。

Microsoft Agent Framework 1.0では、複数の協調パターンもサポートされています。順番に処理する、同時に動かす、別のエージェントへ受け渡す、グループチャットで進める、Magentic-Oneを使う、といった形です。

さらに、本番運用で効いてくる機能も入っています。ストリーミング（途中経過を流しながら返す仕組み）、チェックポイント（途中保存）、人間の承認をはさむ流れ、一時停止と再開などです。

このあたりは、試作では見落としがちですが、本番ではかなり大事です。AIが勝手に突っ走らないようにしたり、失敗したときに途中から戻したり、「ここは人間が確認してから進める」といった制御ができるわけです。

モデルの選択肢も広く、いろいろなプロバイダのモデルにつなげる作りになっています。クラウド上の各種モデルだけでなく、ローカル実行にも対応する方向です。

意味としては、AutoGen単体はもう保守モード。新規に始めるなら、この統合版を見る流れです。ばらばらだった規格がMCP/A2Aに寄って、「ちゃんとつながる」方向へ揃ってきた、と見るとわかりやすいです。

## 数字で見る：性能と料金

![4フレームワークのタスク完了率と料金の目安を比較した表](/images/jp-digest-20260712-ai-agent-framework/figure5.png)

フレームワーク選びで気になるのが、実際どれくらい動くのか、そしていくらかかるのかですよね。

ある比較ブログでは、ローカルでQwen3 32Bというモデルを使い、難易度別に各200タスクを実行したベンチマークが紹介されています。これはあくまで特定の1環境での結果です。その前提で見ると、タスク完了率は次のようになっています。

![CrewAI vs LangGraph vs AutoGen 2026を比較したPooya Golchianの記事](/images/jp-digest-20260712-ai-agent-framework/source1.png)
*出典: Pooya Golchian（https://pooyagolchian.com/blog/crewai-vs-langgraph-autogen-comparison-2026/）*

簡単／中くらい／難しい、の順です。

- LangGraph: 88% ／ 76% ／ 62%
- AutoGen: 85% ／ 68% ／ 58%
- smolagents: 82% ／ 73% ／ 49%
- CrewAI: 79% ／ 71% ／ 54%

この比較では、LangGraphが総じて安定して高めです。特に難しいタスクになるほど、差が見えやすくなっています。

smolagentsは、簡単〜中くらいのタスクではけっこう健闘しています。ただ、難しいタスクでは49%まで落ちていて、複雑な判断や長い手順が絡むと崩れやすい傾向が出ています。

CrewAIも中くらいまでは悪くありませんが、難しいタスクではLangGraphより低めです。AutoGenは全体として中間にいる感じですね。

ただし、ここは本当に大事です。この数字は、特定の1環境・ローカルモデルでの結果です。使うモデル、本番のツール構成、タスクの種類、プロンプトの作り方で変わります。ベンチマークは地図にはなりますが、目的地までの渋滞までは教えてくれません。

料金の目安も、同じ比較ブログで紹介されています。

LangGraphのクラウドは、月49ドルで1デプロイから、月99ドルで5デプロイというプランが示されています。CrewAIは月200回まで無料で、その先は29ドルで1000回、99ドルで5000回という目安です。

AutoGenとsmolagentsはオープンソースで無料、自前で動かす形です。

ただ、「無料＝タダで済む」ではないのがエンジニア世界のこわいところです。自前運用なら、サーバー代、監視、更新、障害対応、セキュリティ対応といった手間がのってきます。財布からは出ていなくても、あなたの週末から出ているかもしれません。つらい。

## で、結局どれを選べばいい？

![状況別にどのフレームワークを選ぶかをまとめた早見表](/images/jp-digest-20260712-ai-agent-framework/figure6.png)

ここまで見てきた通り、AIエージェントの作り方は、ほぼフレームワークの選び方です。なので、状況別にざっくり決めるなら、こんな感じです。

- 本番で「監査ログ・巻き戻し・人間の承認」が要る。失敗をきちんと説明したい、取り消したい  
  → LangGraph

- とにかく速く試作したい。社内ツールやコンテンツ生成の流れ作業を作りたい  
  → CrewAI

- Pythonでデータ処理やコード操作が主役。科学計算や自動スクリプトをやりたい  
  → smolagents  
  ただし、顧客と直接やりとりする画面にはあまり向きません。

- .NETや企業のMicrosoft系基盤に乗っている  
  → Microsoft Agent Framework

- TypeScriptで固めたい  
  → Mastra

- コードを書かずに画面で組みたい  
  → Dify

もう少し補足すると、LangChainは概念を学ぶ入口としては良かった、という声があります。一方で、規模が大きくなるとLangGraphに直接行くか、LangChainの層を外すチームが増えている、という現場の声もあります。

これはLangChainが悪いというより、エージェントが本番に近づくほど、「状態をどう持つか」「失敗時にどう戻すか」「どこで人間を挟むか」が重要になってくるからです。便利な抽象化より、制御しやすさが欲しくなる場面が増えるんですよね。

## 作り始めるときの注意点

最後に、実際に作り始めるときの注意点もまとめておきます。

まず、GitHubスター数などの人気指標は、媒体によってけっこう割れます。時点や数え方で変わるので、鵜呑みにしないほうがいいです。数万スター級なら十分注目されている、くらいの温度感で見てください。

次に、ベンチマークは特定の1環境での結果です。自分が使うモデル、自分の本番条件、自分のタスクで測り直す前提にしたほうが安全です。特にエージェントは、モデルだけでなくツール設計や状態管理でかなり結果が変わります。

CrewAIは手早い一方で、本番では「抽象化が厚くて、LLMに何を送っているか見えづらい」「デバッグが辛い」という利用者の声もあります。試作ではありがたい抽象化が、本番の調査では霧になることがあるんです。

OpenAI Agents SDKやGoogle ADKは、それぞれ特定の提供元のツールに強く結びつきやすいです。つまりロックイン（囲い込み）リスクがあります。最初は便利でも、あとから乗り換えづらくなる可能性は織り込んでおきたいところです。

おすすめの順番は、まず小さく1体作って動かすことです。いきなりマルチエージェントで壮大な社内AI軍団を作ろうとすると、だいたい指揮系統がカオスになります。

最初は、1つのタスクを1体のエージェントにやらせる。  
次に、ツールを1つ足す。  
そのあとで、状態管理や監査、承認フローが必要になった段階でLangGraphなどの本格派へ進む。

このくらいの刻み方が、いちばん失敗しにくいです。

## まとめ

2026年は、AIエージェントのフレームワークが「乱立」から「統合」へ動いた年です。規格はMCP/A2Aへ寄り、どのフレームワークも複数モデル対応へ収束してきています。

だから選ぶ基準は、「人気があるか」よりも「自分の状況に合うか」です。本番か試作か。主役は会話か、データ処理か。使いたい言語や企業基盤は何か。そこから逆算するのが近道です。

まずは小さく作って、実際に動かしてみてください。エージェント開発は、眺めているより触ったほうが一気にわかります。最初の1体が動いた瞬間、フレームワーク選びの霧もかなり晴れてきますよ。
